# Open Questions — проект ПилиСтрогай

Этот файл — единая точка для всех отложенных / неотвеченных вопросов проекта.

## Правила работы с файлом

- Любой открытый вопрос фиксируется здесь с указанием **чата для разрешения**.
- Каждый чат регулярно обращается к этому файлу и спрашивает оператора по своим
  вопросам.
- При закрытии вопроса — статус меняется на `closed`, указывается решение
  (ADR / коммит / промт) и дата закрытия. Вопросы не удаляются сразу — остаются
  в файле с меткой `closed` для истории. Периодически закрытые вопросы
  архивируются блоком в конце файла.

## Формат записи

```markdown
## [YYYY-MM-DD] — короткий заголовок
- **Суть:** формулировка вопроса
- **Контекст:** почему вопрос возник, к какому решению относится
- **Чат:** Архитектурный штаб | Prompt Factory | Настройка Cowork | Триаж MCP | Актуализация KB | другое
- **Приоритет:** high | medium | low
- **Статус:** open | in-progress | closed
- **Решение:** (заполняется при закрытии) ADR-XXX / PR / промт / дата
```

### [2026-04-23] — Дублирование клиентов по telegram_id — политика мержа

- **Суть:** при разборе чата через `link_chat_to_customer` может обнаружиться, что `telegram_id` из чата уже у другого клиента. - **Чат:** Архитектурный штаб (при появлении первой реальной коллизии в операционной работе)

  - `crm-mcp/tools/get_unreviewed_chats.py` (read-tool): возвращает чаты с `review_status='unreviewed'`, sorted by `last_message_at DESC`, с превью первого и последнего текстовых сообщений (≤100 символов, медиа без текста пропускается).   - `crm-mcp/tools/link_chat_to_customer.py` (write-tool): три взаимоисключающих режима — `customer_id` (привязка к существующему), `create_new=True` (создать нового клиента из данных чата), `ignore=True` (пометить как ignored). Атомарная транзакция: всё в одной `session.commit()`, при `SQLAlchemyError` — `session.rollback()`. Pre-validation выбрасывает `ValueError` **до** любых записей в БД.

  - Fallback name при `create_new` с пустым `title`: `"Telegram user {telegram_chat_id}"`.
