# ADR-017: Фильтрация исторических заказов при apply LLM-анализа

**Статус:** принят
**Дата:** 2026-04-30
**Связанные ADR:** ADR-011 §7 (apply analysis), ADR-013 (preflight classification)
**Связанные документы:** `analysis/prompts.py` (NARRATIVE_PROMPT, STRUCTURED_EXTRACT_PROMPT), `app/analysis/service.py::apply_analysis_to_customer`

---

## Контекст

Промт `NARRATIVE_PROMPT` (v1.4) by design просит Qwen3-14B реконструировать **всю историю заказов клиента** из переписки — включая завершённые, отменённые и возвращённые. `STRUCTURED_EXTRACT_PROMPT` (v1.4) превращает narrative-портрет в JSON `extract.orders[]`. На стороне применения `apply_analysis_to_customer` (ADR-011 §7) трактует **каждый элемент `extract.orders` как новый draft-заказ** для оператора. Контракт на стороне промта и контракт на стороне apply не совпадают.

**Smoke-кейс chat 6485 (Слава Яшин, customer_id=1688), 2026-04-30:**

- v1.4 выдал 26 orders в `structured_extract.orders`.
- 23 из 26 — без items (исторические упоминания «оформил заказ», «доставлено», «возврат Veritas MK-II»).
- 3 из 26 — с items, в том числе позиции с `matching_status='not_found'`.
- `apply_pending_analysis(6485)` создал 26 draft-обёрток в `orders_order` (большинство пустых).
- Повторный вызов с `force=True` создал ещё 26 — всего 50 пустых draft. Журнал `analysis_created_entities` обнулился через `delete_created_entities`, но **сами `orders_order` остались** (force=True не каскадирует, см. E18).

**Статистика по уже прогнанным v1.4 чатам (4 чата, 36 orders на 2026-04-30):**

- `orders_with_items=13 (36.1%)`
- `orders_empty=23 (63.9%)`
- Терминальные статусы у пустых: 19/23 имеют `unknown/unknown`, 3/23 — `returned/*`. То есть статус сам по себе **не является надёжным сигналом** «исторический»: Qwen в большинстве случаев не выводит финальные значения статусов, даже когда заказ из narrative явно завершён.

**Архитектурный диагноз.** Бага в коде apply нет — он работает по своему контракту. Промт работает по своему. Несовпадение лежит на стыке: extract.orders семантически — это «всё, что упоминается в чате», а apply ожидает «новые черновики, которые нужно довести до confirmed». Оператор получает мусор: пустые draft-обёртки заполняют Cowork-листинг, реальные действующие заказы теряются среди них.

**Что не обсуждается в этом ADR:** содержимое промта v1.4 (его не меняем — выкинутые orders остаются доступны для read-only анализа через `analysis_chat_analysis.structured_extract`); схема `MatchedStructuredExtract` (флаг `is_active` отвергается, см. альтернативы); реализация фильтра в коде (это G4.6).

---

## Решение

В `apply_analysis_to_customer` (`app/analysis/service.py:412`) добавляется фильтр `_is_actionable_order(order: MatchedOrder) -> bool` перед циклом создания заказов. Order создаётся в `orders_order` **только если** выполнены оба условия:

1. **`order.items` непустой и содержит хотя бы один MatchedOrderItem** (любого `matching_status` — `confident_match`, `ambiguous`, `not_found`).
   *Обоснование.* Это основной защитник: 23/23 пустых orders в smoke-данных режутся. Если LLM смог распарсить позиции — это сигнал, что заказ обсуждался конкретно, а не упомянут вскользь («оформил как обычно»). `not_found` items также пропускаются — они формируют pending_items для оператора, что является валидной частью workflow.

2. **`order.status_delivery` ∉ `{'delivered', 'returned'}`.**
   *Обоснование.* Вторичный фильтр для оставшихся ~36% orders с items. На текущей выборке режет дополнительно 3/23 (`returned`), на массовой выборке доля может вырасти. Дёшево, очевидно, без false-negative для реальных action-кейсов.

**Дополнительное условие НЕ применяется:** `status_payment='paid' AND status_delivery='shipped'` отвергнуто — `shipped` означает «в пути», заказ ещё не закрыт, оператор может захотеть сопровождать его в Cowork.

**Что происходит с отфильтрованными orders.** Полностью игнорируются на стороне apply: НЕ создаются ни `orders_order`, ни pending_items, ни journal-записи `analysis_created_entities`. Они остаются read-only в `analysis_chat_analysis.structured_extract` и доступны через будущий MCP-tool `get_customer_history` (см. `tool-gaps.md`).

**Счётчики apply** дополняются: `AnalysisApplicationResult.orders_filtered_historical: int` — сколько orders прошли мимо фильтра. Оператор видит это в выдаче `apply_pending_analysis`: `"Заказы: создано=N, позиций=M, отфильтровано как история=K"`.

## Альтернативы рассмотрены

**(B) Флаг `is_active: bool` в `Order` schema + bump промта v1.5.**
LLM сам отмечает «активный» vs «исторический» на основе narrative.

- ❌ Требует bump `ANALYZER_VERSION` → новый прогон 386 чатов на PC (часы + тепловой цикл).
- ❌ LLM-флаг — эвристика, не лучше детерминированных правил из текущей выборки (где 19/23 имеют `unknown/unknown` — Qwen и так не очень уверенно проставляет статусы).
- 🟡 Откладывается до триггера: если фильтр (A) режет ≥10% реально нужных orders в первой партии 386 чатов, бампится промт. Записано в `06_open_questions.md`.

**(C) Новая таблица `orders_historical_order` для read-only истории.**
Отдельное хранилище без триггеров статусов, без procurement.

- ❌ Избыточно для read-only данных, которые уже лежат в `analysis_chat_analysis.structured_extract`.
- ❌ Миграция Alembic, новые модели, новый MCP-tool — overkill для текущего объёма.
- ❌ Дублирование данных между JSONB и таблицей.

**(D) Фильтр на стороне промта STRUCTURED_EXTRACT.**
Просить LLM не выводить historical orders в JSON.

- ❌ Снова требует bump v1.5 + перепрогон.
- ❌ Теряем reconstruction чата для будущих read-only кейсов (анализ предпочтений по истории покупок).

## Последствия

**Положительные:**

- G4 разблокируется без миграций и без перепрогона PC.
- Cowork не получает мусора при массовом apply 386 чатов — оператор видит только actionable drafts.
- `force=True` reapply не множит мусор (orders, не прошедшие фильтр, и так не создаются).
- История заказов клиента сохраняется в `structured_extract` для будущего read-only доступа.

**Отрицательные / компромиссы:**

- Оператор теряет автоматическое создание draft-ов для orders, у которых LLM не смог распарсить items (~64% по текущей выборке). Если это были действительно новые активные заказы без структурных данных — оператор должен завести их вручную через `create_order` после прочтения narrative. Митигация: `tool-gaps.md` запись на `get_customer_history` для удобного доступа к narrative.
- Фильтр **детерминирован**, но обрезает по жёстким правилам. Edge case: LLM выдаёт `status_delivery=delivered` для заказа, который реально ещё в работе (ошибка LLM) → draft НЕ создаётся. Митигация: триггер на bump промта (см. `06_open_questions.md`).
- ADR-011 §7 не противоречит этому решению, но получает уточнение: «не каждый элемент `extract.orders` материализуется в `orders_order`».

**Триггер для следующего ADR:**
По итогам массового прогона 386 чатов оператор оценивает долю «упущенных нужных orders» (LLM правильно нашёл активный заказ, но фильтр его срезал). Если ≥10% — открывается ADR-018 на bump промта v1.5 с явным флагом `is_active`.
