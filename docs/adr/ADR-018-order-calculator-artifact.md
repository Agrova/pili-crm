# ADR-018: Калькулятор нового заказа как первый write-артефакт

**Статус:** принят
**Дата:** 2026-05-05
**Связанные ADR:** ADR-001 v2 (модульный монолит, Cowork+MCP как интерфейс), ADR-003 final-ready + Addendum statuses + Addendum quoted, ADR-004 (pricing & profit policy), ADR-016 (token economy + mobile capture), ADR-F04 (snapshot цен), ADR-F06 (хранение валюты)
**Аддендумы:** [ADR-018 Addendum: Политика обновления курса валют через калькулятор](ADR-018-addendum-exchange-rate-update-policy.md) (2026-05-06) — уточняет развилку C и детализирует поведение «правка курса в калькуляторе» (двойное подтверждение, отсутствие черновой памяти, отдельная запись в `pricing_exchange_rate` через явное согласие).
**Связанные документы:** `docs/cowork-system-prompt.md`, `docs/PLAN.md` (группа G19), `docs/schema-gaps.md` (запись 2026-05-05 про уникальные товары)

## Контекст

ADR-016 G18 решает экономию токенов на read-операциях через Cowork live artifacts (три read-only дашборда). Параллельно у оператора возникла конкретная боль, которую read-only артефактом не закрыть:

**Боль — расчёт нового заказа из нескольких товаров занимает много шагов в Cowork.**

Оператор делает следующее (текущий путь):
1. Спрашивает Cowork «какой курс USD сейчас» → токены.
2. «Найди клиента Воропаев» → токены.
3. Если нет — «создай клиента» → токены + правило двух подтверждений.
4. По каждому товару: «найди в каталоге Veritas Shooting Board» → «возьми его цену закупки и вес» → «посчитай цену клиенту с маржой 30%» → токены × N товаров.
5. «Сформируй текст заказа для отправки клиенту в Telegram» → токены.
6. Копирует, отправляет в Telegram.
7. Через 2 дня клиент согласился — оператор: «создай заказ из того расчёта» → токены + правило двух подтверждений.

На один заказ из 3 позиций — десятки тысяч токенов и 10+ ходов в чате. При двух заказах в день — это значимая часть месячной квоты.

**Дополнительные сложности, которые путь через чат маскирует:**

- **Уникальные товары** (eBay-разовые, штучные книги) — не каталожные, оператор каждый раз руками вбивает вес/цену. Нет места в схеме (см. `docs/schema-gaps.md` запись 2026-05-05).
- **Курс может быть устаревшим** — оператор иногда замечает только постфактум, что считал по вчерашнему курсу.
- **Шаблон сообщения клиенту** — оператор каждый раз заново формулирует «здравствуйте, по вашему запросу...». Нет места хранить шаблон.
- **Заказ существует в подвешенном состоянии** между «отправили клиенту» и «клиент согласился» — нет статуса (решено addendum-ом про `quoted`).

**Что не обсуждается в этом ADR:**
- Полный мобильный CRM (G14 — отдельный ADR при необходимости).
- Read-only артефакты G18.1 (отдельная задача).
- Решение по уникальным товарам — это отдельный мини-ADR, см. open questions.

## Варианты

### Развилка A — где живёт калькулятор

#### Вариант A1 — Cowork live artifact с write-возможностью ✅

Артефакт `order_calculator.html` через `mcp__cowork__create_artifact`. Использует `window.cowork.callMcpTool` для всех операций включая write (`create_order(status='quoted')`).

- **Плюсы:** реиспользует существующий механизм (как G18.1 read-артефакты); ноль внешней инфры; персистент через сессии; написание через один промт Prompt Factory; естественное продолжение ADR-016.
- **Минусы:** **смещает архитектурную границу ADR-001 v2** («Cowork+MCP — единственный интерфейс») — write идёт через артефакт, не через Cowork-диалог; правило двух подтверждений не применяется автоматически (артефакт показывает превью, кнопка «создать» — это и есть подтверждение).

#### Вариант A2 — Cowork-диалог с подсказками от Cowork

Оставить как есть, улучшить только Cowork-промт чтобы он быстрее проводил оператора через шаги.

- **Плюсы:** не меняем архитектуру.
- **Минусы:** не решает корневую проблему — токены тратятся; UX лучше не становится.

#### Вариант A3 — Standalone web-app на FastAPI + Jinja

Отдельная страница `localhost:8000/calculator`. Не зависит от Cowork.

- **Плюсы:** независимо от Cowork; полная свобода UI.
- **Минусы:** разработка фронта; дублирование с механизмом артефактов; нужен auth (хоть для одного оператора); противоречит «разумной достаточности» ADR-016.

### Развилка B — где считается цена

#### Вариант B1 — Сервер через MCP-tool `calculate_price` ✅

Артефакт собирает входные данные (клиент, состав, курс), вызывает `calculate_price(items=[...], exchange_rate=...)`, сервер возвращает полный расчёт по правилам ADR-004.

- **Плюсы:** один источник истины; pricing-формула остаётся в Python; артефакт нельзя «обмануть» правкой JS; полностью консистентно с будущим apply.
- **Минусы:** требует создать новый MCP-tool `calculate_price` (его нет среди 19 существующих).

#### Вариант B2 — JS в артефакте

Pricing-формула продублирована в JavaScript.

- **Плюсы:** мгновенный пересчёт без round-trip.
- **Минусы:** **нарушает ADR-004** (pricing-константы только через Prompt Factory); два источника истины; формула усложнится — JS отстанет.

#### Вариант B3 — JS прикидывает, сервер фиксирует при сохранении

Артефакт показывает оценку в JS для интерактивности, при «создать» — пересчёт на сервере.

- **Плюсы:** живой UX.
- **Минусы:** оператор может увидеть в превью одну цену, в созданном заказе — другую. Психологически плохо.

### Развилка C — курс валюты

#### Вариант C1 — Артефакт читает текущий курс из БД, рядом кнопка «обновить» ✅

Артефакт при загрузке вызывает `get_current_exchange_rate('USD')`, показывает значение и дату последнего обновления. Если оператор видит, что курс устарел — кнопка «обновить курс» открывает мини-форму, вызывает `set_exchange_rate('USD', new_rate)`. Расчёт **всегда** по курсу из БД.

- **Плюсы:** один источник истины (БД); нет молчаливых расхождений; явная ручка для случая «устарел»; зависимость от G17 (`get_current_exchange_rate` / `set_exchange_rate`) ясная.
- **Минусы:** требует G17 (или его части) до запуска G19.

#### Вариант C2 — Оператор вводит курс в калькуляторе на разовое использование

Расчёт по введённому курсу, в БД ничего не пишется.

- **Минусы:** курс в калькуляторе и в БД могут разъехаться; при создании `quoted` запись будет с курсом из БД (или придётся передавать override) — неконсистентно.

### Развилка D — товары не из каталога (уникальные)

#### Вариант D1 — Inline-описание в order_item ✅ (предварительно)

Расширить `order_item` полями для уникальных товаров: `product_id NULLABLE`, `inline_name`, `inline_weight_kg`, `inline_purchase_price_fcy`, `inline_purchase_currency`, `inline_source_url`.

- **Плюсы:** компактно, минимальная миграция; уникальный товар «живёт» в заказе и больше нигде; не засоряет каталог.
- **Минусы:** колонки на уровне `order_item` дублируют поля каталога — небольшое нарушение нормализации; CHECK constraint сложнее.

#### Вариант D2 — Отдельная таблица `orders_unique_purchase`

FK от `order_item.unique_purchase_id`, сама таблица хранит inline-описание.

- **Плюсы:** чище семантически; легко расширять (добавить статус «получено», ссылку на shipment).
- **Минусы:** новая таблица, новые миграции, новый репозиторий и схемы.

#### Вариант D3 — Каталог с флагом `is_unique=true`

Все товары в каталоге, уникальные помечены флагом, скрыты из обычного поиска.

- **Минусы:** каталог растёт мёртвыми SKU; нужна логика «забыть» уникальный товар; противоречит решению оператора «их не надо заносить в каталог».

**Решение по развилке D отложено в отдельный мини-ADR** (см. `docs/schema-gaps.md` запись 2026-05-05). G19.0 (этот ADR) фиксирует **функциональность калькулятора** независимо от того, как технически решится хранение уникальных товаров — артефакт должен работать с любым из вариантов D1/D2 после реализации миграции.

### Развилка E — шаблон сообщения клиенту

#### Вариант E1 — В БД (новая мини-сущность `message_templates`) ✅

Таблица `message_templates(id, code, body_template, language, is_active, updated_at)`. Калькулятор берёт активный шаблон с `code='quote_to_client'`, подставляет переменные (клиент, состав, цена, курс).

- **Плюсы:** оператор может править шаблон без правки артефакта; версионируется; в будущем — несколько шаблонов (B2C / B2B / по категории товара).
- **Минусы:** новая таблица + базовый шаблон в seed.

#### Вариант E2 — Шаблон в коде / артефакте

Жёстко зашит в HTML или Python-файле.

- **Минусы:** правка = редактирование артефакта или релиз; никакой A/B; никаких вариантов.

## Принятое решение

### Архитектурный профиль

| Развилка | Принято |
|---|---|
| A — где живёт | **A1** — Cowork live artifact с write |
| B — где считается цена | **B1** — Сервер через `calculate_price` MCP-tool |
| C — курс | **C1** — Из БД с кнопкой «обновить» (политика обновления — см. [Addendum: Политика обновления курса валют](ADR-018-addendum-exchange-rate-update-policy.md), 2026-05-06) |
| D — уникальные товары | **отложено в мини-ADR** — артефакт работает с D1 или D2 после реализации |
| E — шаблон сообщения | **E1** — `message_templates` таблица |

### Явное смягчение принципа ADR-001 v2

ADR-001 v2 фиксирует «Cowork + MCP — единственный пользовательский интерфейс». ADR-016 уже сделал шаг от read-only артефактов (артефакты тоже UI, но read). ADR-018 идёт дальше: **артефакт-калькулятор пишет в БД через MCP-tool**.

Это сознательное смягчение принципа, ограниченное:
- **Только статус `quoted`** — артефакт не создаёт `confirmed`, `in_procurement` и далее. Перевод в `confirmed` — только через Cowork.
- **Только функция «новый заказ»** — артефакт не редактирует существующие заказы, не отменяет их, не меняет статусы у других сущностей.
- **Только один артефакт-калькулятор** — любой следующий write-артефакт = новый ADR с обоснованием.

Этот принцип фиксируется как новый инвариант: **граница «write через артефакт» не расширяется без отдельного ADR**.

### Поток работы

1. Оператор открывает калькулятор (любой Cowork-чат: «открой калькулятор нового заказа»).
2. Артефакт при загрузке: `get_current_exchange_rate('USD')` + базовый шаблон сообщения.
3. Оператор ищет клиента → `find_customer(query=...)`. Если нет — «новый» открывает форму, вызывает `create_customer` (это write через артефакт — ОК, в рамках сценария «новый заказ»).
4. Оператор добавляет позиции:
   - **Из каталога** — `search_products(query=...)`, выбор → артефакт получает `weight_kg`, `purchase_price_fcy`, `purchase_currency`, `inline=false`.
   - **Уникальный** — оператор заполняет `inline_name`, `inline_weight_kg`, `inline_purchase_price_fcy`, `inline_purchase_currency`, опционально `inline_source_url`.
5. После каждого добавления / изменения позиции — `calculate_price(items=[...], exchange_rate=89.50)` возвращает по позиции и итог.
6. Текст для клиента генерируется автоматически по шаблону, оператор может его подкорректировать.
7. Оператор:
   - **«Скопировать»** — текст в буфер, ничего не сохраняется. Полезно когда клиент явно «прицениться».
   - **«Создать quoted и скопировать»** — `create_order(customer_id=..., items=[...], status='quoted', client_message=...)` → возвращает `order_id`, текст в буфер. Запись в БД зафиксирована.
8. Оператор отправляет текст в Telegram клиенту (вне артефакта).
9. Через сколько-то дней:
   - Клиент согласился → оператор в Cowork: `confirm_quote(order_id)` или явно `update_order(status='confirmed')` → derive-trigger переводит в `in_procurement`.
   - Клиент отказался / молчит → оператор в Cowork: `cancel_order(order_id)`.

### Защита и инварианты

- **Артефакт не может перевести заказ дальше `quoted`.** В `create_order` — параметр `status` принимает только `draft` или `quoted`, любое другое значение из артефакта — ошибка на уровне MCP-tool.
- **Snapshot цены (ADR-F04).** При создании `quoted` `pricing_price_calculation` сохраняется как обычно — клиент может ссылаться на «цену из вчерашнего расчёта» и она зафиксирована.
- **Иммутабельность after `quoted`.** Если оператор хочет «пересчитать с новым курсом» — это не редактирование, а **новая** quoted (старая отменяется или остаётся с пометкой `superseded_by=...` — пока без поля, оператор отменяет старую руками).
- **Идемпотентность артефакта.** Двойной клик «создать quoted» не должен создать два заказа. Артефакт блокирует кнопку после первого клика, дожидается ответа MCP, в случае ошибки — разблокирует.

## Последствия

### Что становится проще

- Расчёт нового заказа из 3 позиций — 30 секунд в калькуляторе вместо 5 минут в чате с десятками тысяч токенов.
- Шаблон сообщения клиенту обновляется централизованно через `message_templates`, не правя артефакт.
- Курс всегда из БД, оператор видит дату последнего обновления — больше не считает по вчерашнему курсу молча.
- Уникальные товары получают официальное место в схеме (после реализации D1 или D2), а не в комментариях к заказу.
- Статус `quoted` явно отражает «отправили, ждём» — не путается с черновиками в работе.

### Какие ограничения появляются

- Артефакт-калькулятор — **первый write-артефакт в проекте**. Прецедент для будущих расширений: каждый следующий write-артефакт требует ADR.
- Calculate_price MCP-tool становится центральной точкой pricing — изменения политики ADR-004 теперь идут через этот tool, а не через изолированные функции.
- Шаблон сообщения клиенту в БД — оператору надо помнить, что правка идёт там, не в артефакте. Решается описанием в runbook.
- При устаревании курса оператор должен **сам** обновить — кнопка есть, но автообновления нет (это намеренно — курс ставит человек, не автомат).

### Что придётся учитывать дальше

- Как только реализован `calculate_price` — прежние pricing-функции в коде (если есть отдельные) консолидируются в его реализацию. Иначе два источника правды.
- Через 1-2 месяца работы калькулятора — оценка статистики `quoted`-заказов. Сколько подтвердили, сколько отменили, сколько висят. Это даст материал для решения по TTL (см. ADR-003 addendum quoted, открытые вопросы).
- Если оператор начнёт просить «отредактировать quoted перед отправкой клиенту» — нужна функция «удалить и пересчитать заново», без in-place редактирования.
- При появлении G18.1 (read-only артефакты `pending_orders_dashboard`) — добавить фильтр по статусу `quoted`, чтобы оператор видел свои висячие предложения.

## Что должен сделать Claude Code

Реализация — отдельная задача G19, разбита на подзадачи. Здесь зафиксированы только обязательные точки.

### Подзадача G19.1 — `calculate_price` MCP-tool

- Файл `crm-mcp/tools/calculate_price.py`.
- Сигнатура:
  ```python
  calculate_price(
      items: list[CalculatePriceItem],   # каталожные ИЛИ inline-описание
      exchange_rate: Decimal | None = None,  # None = взять из БД
      customer_id: int | None = None,    # для будущих скидок по клиенту
  ) -> CalculatePriceResponse
  ```
- Возвращает по каждой позиции: `purchase_price_rub`, `markup_amount_rub`, `final_price_rub`, `weight_shipping_rub`, `profit_estimate_rub`, `profit_percent`. И итог.
- Использует существующие pricing-функции из `app/pricing/` — никакой логики не дублирует.
- 8-10 тестов: пустой список, одна позиция, несколько, каталожная + inline, override курса, разные валюты.

### Подзадача G19.2 — миграция и модели для `quoted`

См. ADR-003 addendum про `quoted` (отдельный документ). Включает:
- ALTER TYPE для enum.
- Обновление derive-trigger.
- `confirm_quote(order_id)` MCP-tool или расширение `update_order`.

### Подзадача G19.3 — миграция для уникальных товаров

Реализуется по результату мини-ADR (D1 или D2). Не блокирует G19.1/G19.2 (калькулятор может быть выпущен в режиме «только каталожные» как первая версия), но **блокирует** полную функциональность.

### Подзадача G19.4 — `message_templates` + базовый шаблон

- Миграция: таблица `message_templates(id, code VARCHAR PK, body_template TEXT, language VARCHAR(5), is_active BOOL, updated_at)`.
- Seed: `code='quote_to_client'`, базовый русский шаблон (см. примерный текст в разделе «Поток работы»).
- MCP-tool `get_message_template(code, language='ru')` — read.
- Опционально (отложить): MCP-tool `update_message_template` — для редактирования через Cowork.

### Подзадача G19.5 — артефакт `order_calculator.html`

- Файл `crm-mcp/artifacts/order_calculator.html`.
- Использует `window.cowork.callMcpTool` для всех вызовов: `find_customer`, `create_customer`, `search_products`, `get_current_exchange_rate`, `set_exchange_rate`, `calculate_price`, `get_message_template`, `create_order`.
- Состояние клиента/состава/курса — в JS памяти (не localStorage, чтобы не путать недоделанные расчёты между сессиями).
- Блокировка кнопки «создать quoted» при первом клике.
- Превью текста — `<textarea>` с дефолтным содержимым из шаблона, оператор может править перед копированием.
- 4-5 ручных smoke-сценариев в runbook (создание новой quote с каталожным товаром / с уникальным / с двумя / с обновлением курса / с новым клиентом).

### Подзадача G19.6 — обновление `cowork-system-prompt.md`

Добавить раздел «Калькулятор нового заказа» — когда Cowork предлагает оператору открыть артефакт (вместо проведения расчёта в чате), как реагировать на «созданный quoted-заказ» (в pending-обзоре он должен показываться отдельно), правила для перехода `quoted → confirmed` через `confirm_quote`.

## Открытые вопросы

- **Q (для архитектурного штаба, до запуска G19):** мини-ADR по развилке D (уникальные товары) — D1 inline-поля vs D2 отдельная таблица. Без этого решения G19.5 артефакт может быть выпущен только с каталожными товарами, что закрывает 60% потребности оператора, но не 100%.

- **Q (для архитектурного штаба, после запуска G19):** TTL для `quoted` (см. ADR-003 addendum quoted). Сколько `quoted`-заказов накопится за 1-2 месяца и сколько из них реально «висят» — это ответ.

- **Q (для оператора, после запуска G19):** нужна ли функция «пересчитать quoted с новым курсом» (создаёт новый quoted, старый автоматически в `cancelled` с reason='superseded_by_quote_NNN')? Активация — реальный запрос «клиент не ответил неделю, курс уехал, надо пересчитать».

- **Q (для оператора, после запуска G19):** нужны ли разные `message_templates` для разных типов клиентов / категорий товаров? Активация — несколько раз руками меняешь текст после генерации.
