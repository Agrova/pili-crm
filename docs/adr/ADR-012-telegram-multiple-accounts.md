# ADR-012: Поддержка нескольких Telegram-аккаунтов

**Статус:** draft
**Дата:** 2026-04-24
**Связанные ADR:** ADR-003 final-ready (core schema), ADR-009 (Telegram profile schema), ADR-010 (Telegram ingestion pipeline), ADR-011 (Telegram chat analysis pipeline)
**Связанные документы:** `06_open_questions.md`, `ingestion/tg_import.py`

---

## Контекст

Текущая схема `communications_telegram_chat` (ADR-003 + ADR-009 + ADR-010) построена в предположении **одного Telegram-аккаунта оператора**. Уникальный ключ `uq_communications_telegram_chat_telegram_chat_id UNIQUE (telegram_chat_id)` различает чаты только по id в рамках одного аккаунта.

Операционная реальность отличается:

- Оператор ведёт переписку **с двух телефонов**, с двумя разными номерами и двумя разными Telegram-аккаунтами:
  - `+77471057849` — казахстанский (выгрузка `DataExport_2026-04-11`, уже импортирована, 303 чата, 64 785 сообщений)
  - `+79161879839` — российский (выгрузка есть, пока не импортирована)
- Клиенты пишут на оба аккаунта. Один физический клиент — **один человек** (один глобальный `telegram_id`), но может быть доступен через **два разных чата** (по одному в каждом аккаунте).
- Операционный принцип оператора: по возможности вести переписку с одного номера, но это не всегда получается — клиент сам выбирает, куда написать.

**Проблема 1 — коллизия `telegram_chat_id`.** Идентификатор чата генерируется Telegram независимо в каждом аккаунте. Поля `telegram_chat_id=12345` из аккаунта 1 и аккаунта 2 — это разные чаты с разными людьми. Текущий UNIQUE constraint при импорте второй выгрузки схлопнет разные чаты в один:

- Первая выгрузка: чат «Максим» с `telegram_chat_id=12345` импортирован → `communications_telegram_chat.id=100`
- Вторая выгрузка: чат «Константин» с `telegram_chat_id=12345` (совпадение id внутри другого аккаунта) → `INSERT ... ON CONFLICT (telegram_chat_id) DO NOTHING` срабатывает, запись не создаётся
- **Все сообщения Константина импортируются с FK на `chat_id=100`** — то есть попадают в чат Максима. Данные невосстановимо перемешиваются.

**Проблема 2 — один клиент, несколько чатов.** Клиент Иван пишет и на казахстанский, и на российский аккаунт. В текущей схеме `orders_customer_profile` и MCP-tool `link_chat_to_customer` предполагают связь «один клиент — один Telegram-чат» (через `communications_telegram_chat.customer_id`). Связать второй чат с тем же клиентом в текущей модели можно — но нет способа отличить «это тот же Иван, пишет с другого канала» от «это два разных Ивана, совпали имена».

**Проблема 3 — процесс импорта не знает, какому аккаунту принадлежит выгрузка.** `ingestion/tg_import.py` читает JSON-выгрузку Telegram Desktop и записывает чаты. Никакой метки «эта выгрузка из аккаунта X» в базе не появляется.

**Решение нужно принять сейчас**, до импорта второй выгрузки и до реализации ADR-011 Задачи 2 (сервисный слой анализатора). Если сначала сделать Задачу 2 на текущей схеме, а потом чинить — придётся рефакторить только что написанный сервис.

## Варианты

### Вариант A — Расширенный композитный ключ + справочник аккаунтов + структура папок по номеру ✅ выбран

Ввести таблицу `communications_telegram_account` как справочник аккаунтов. Каждый чат принадлежит одному аккаунту через FK. UNIQUE ослабляется с `(telegram_chat_id)` до `(owner_account_id, telegram_chat_id)`. Выгрузки на диске раскладываются в папки `/Users/protey/pili-crm-data/tg-exports/<phone_number>/DataExport_*/`, скрипт автодетектит аккаунт по имени папки. Один клиент может быть привязан к нескольким чатам (по одному на каждый аккаунт) — связь через существующее поле `customer_id` в `communications_telegram_chat`, никаких изменений в `orders_customer`.

- **Плюсы:** правильная нормализация; явно отражает операционную реальность; минимум изменений в других модулях; переиспользует confidence-модель ADR-009; структура папок устраняет ручной ввод `--account-phone`.
- **Минусы:** требует миграции существующих 303 чатов с назначением owner_account; перенос существующей выгрузки в подпапку (разовый шаг, уже выполнен оператором 2026-04-24).

### Вариант B — Префикс в `telegram_chat_id` по аккаунту

Менять формат `telegram_chat_id` при импорте: вместо голого id писать `{account_id}:{telegram_chat_id}`. Схема не меняется вообще.

- **Плюсы:** минимум миграции.
- **Минусы:** загрязняет семантику поля; сломает любой код, работающий с `telegram_chat_id` как с числовым id (deep links, связь с raw JSON); невозможно различить «чат из аккаунта 1» и «чат из аккаунта 2» без парсинга строки; неочевидно для будущих разработчиков и Claude Code.

### Вариант C — Две параллельные таблицы `communications_telegram_chat_account1`, `communications_telegram_chat_account2`

Физическое разделение чатов по аккаунту на уровне таблиц.

- **Плюсы:** жёсткая изоляция.
- **Минусы:** ломает все существующие запросы; требует шаблонирования/UNION во всех MCP-tools; не масштабируется; антипаттерн реляционного дизайна.

---

## Критерии выбора

- **Надёжность:** дизайн должен защитить от случайного схлопывания чатов из разных аккаунтов
- **Простота поддержки:** одна понятная таблица-справочник + FK лучше, чем строковые хаки
- **Совместимость с ADR-009 и ADR-011:** не должен ломать JSONB-схему профиля, confidence-модель, уже реализованный анализатор (Задача 1 ADR-011)
- **Сохранение PostgreSQL как источника истины:** телефон аккаунта и metadata должны быть в БД, не в конфиге скрипта
- **Операционная простота:** оператор должен явно знать, какому аккаунту соответствует выгрузка — через структуру папок, без дополнительных параметров CLI в штатном случае
- **Масштабируемость:** оператор не планирует третий аккаунт, но архитектура не должна закрывать такую возможность

## Принятое решение

**Вариант A — справочник аккаунтов + композитный ключ + структура папок по номеру.**

### 1. Новая таблица `communications_telegram_account`

Справочник аккаунтов оператора. MVP — два аккаунта, архитектура не ограничивает N.

| Поле | Тип | Назначение |
|---|---|---|
| `id` | `BIGINT PK` | Суррогатный ключ |
| `phone_number` | `TEXT NOT NULL UNIQUE` | Номер телефона аккаунта в формате E.164 (`+77471057849`). Главный идентификатор — стабилен, есть в JSON-выгрузке в `personal_information.phone_number`, есть в имени папки выгрузки. |
| `display_name` | `TEXT NOT NULL` | Человекочитаемая метка для оператора и отчётов. Значения: `Казахстан (+77471057849)`, `Россия (+79161879839)`. |
| `telegram_user_id` | `TEXT NULL` | Собственный user_id оператора в этом аккаунте (есть в `personal_information.user_id` в JSON). Заполняется автоматически при первом импорте. Полезен для диагностики. |
| `first_import_at` | `TIMESTAMPTZ NULL` | Момент первого импорта выгрузки этого аккаунта. Заполняется при первом прогоне `tg_import`. |
| `last_import_at` | `TIMESTAMPTZ NULL` | Момент последнего импорта. Заполняется при каждом прогоне. |
| `notes` | `TEXT NULL` | Свободные заметки оператора. |
| `created_at`, `updated_at` | `TIMESTAMPTZ` | Стандартные timestamps ADR-003. |

Constraints:
- UNIQUE `phone_number` (имя `uq_communications_telegram_account_phone`)
- CHECK `phone_number ~ '^\+[1-9]\d{7,14}$'` (E.164)

### 2. Изменения в `communications_telegram_chat`

Добавить поле:

| Поле | Тип | Назначение |
|---|---|---|
| `owner_account_id` | `BIGINT NOT NULL FK → communications_telegram_account(id) ON DELETE RESTRICT` | К какому аккаунту оператора принадлежит чат |

`ON DELETE RESTRICT` — удалить аккаунт нельзя, пока у него есть чаты. Это безопасность: аккаунт — идентичность, терять связь чатов с их владельцем нельзя.

Ослабление UNIQUE-constraint:

- **Удалить:** `uq_communications_telegram_chat_telegram_chat_id` (на `telegram_chat_id`)
- **Создать:** `uq_communications_telegram_chat_owner_telegram_chat_id` — UNIQUE `(owner_account_id, telegram_chat_id)`

Никакие другие поля не меняются.

### 3. Изменения в `communications_telegram_message`

**Ничего.** Сообщения связаны с чатом через `chat_id FK → communications_telegram_chat(id)`. Чат уже несёт владельца. Денормализация `owner_account_id` в сообщениях не нужна — запросы, которым нужен аккаунт сообщения, идут через JOIN.

### 4. Миграция существующих данных

Миграция пишется в одну транзакцию:

1. Создать таблицу `communications_telegram_account`
2. INSERT первой записи (казахстанский аккаунт — владелец уже импортированных 303 чатов):
   ```sql
   INSERT INTO communications_telegram_account
     (phone_number, display_name, notes, first_import_at, last_import_at)
   VALUES
     ('+77471057849', 'Казахстан (+77471057849)',
      'Первый импортированный аккаунт (ADR-010 Задание 1, 2026-04-23)',
      '2026-04-23 15:48:00+00', '2026-04-23 15:48:00+00')
   RETURNING id;
   -- expected id=1
   ```
3. Добавить в `communications_telegram_chat` колонку `owner_account_id BIGINT` (пока NULL, не NOT NULL)
4. Backfill: `UPDATE communications_telegram_chat SET owner_account_id = 1 WHERE owner_account_id IS NULL`
5. Проверить: `SELECT COUNT(*) FROM communications_telegram_chat WHERE owner_account_id IS NULL` должен вернуть 0
6. Сделать `owner_account_id NOT NULL`
7. Добавить FK constraint с `ON DELETE RESTRICT`
8. Удалить старый UNIQUE на `telegram_chat_id`
9. Добавить новый UNIQUE на `(owner_account_id, telegram_chat_id)`

Второй аккаунт (`+79161879839`) создаётся **вручную оператором** перед импортом второй выгрузки — через новый CLI-хелпер `ingestion/register_account.py`. В миграцию seed'ом не попадает.

### 5. Структура папок выгрузок на диске

Корневая директория: `/Users/protey/pili-crm-data/tg-exports/`. Раскладка по аккаунтам:

```
/Users/protey/pili-crm-data/tg-exports/
├── +77471057849/                        ← казахстанский аккаунт
│   ├── chats/                           ← плоская структура (предпочтительная)
│   ├── profile_pictures/
│   ├── result.json
│   └── DataExport_2026-04-11/           ← legacy-структура (допустимо, для первой выгрузки)
│       ├── result.json
│       └── ...
└── +79161879839/                        ← российский аккаунт
    ├── chats/                           ← плоская структура
    ├── profile_pictures/
    └── result.json
```

**Правила:**
- Имя папки-аккаунта = `phone_number` в E.164 формате с лидирующим `+`
- **Предпочтительная структура — плоская:** `result.json` и `chats/` лежат прямо в `+PHONE/`, без промежуточной папки с датой. Соответствует тому, как Telegram Desktop выгружает по умолчанию в новых версиях.
- **Legacy-структура (`+PHONE/DataExport_YYYY-MM-DD/result.json`)** — допустимо для уже существующих выгрузок (например, `+77471057849/DataExport_2026-04-11/`). Скрипт поддерживает обе структуры (см. раздел 6).
- **Одна выгрузка на аккаунт.** Новая выгрузка заменяет старую (см. раздел 10 «Политика хранения выгрузок»). Исторические `DataExport_*` папки в новой парадигме не накапливаются.

**Операционный перенос существующей выгрузки (однократный, уже выполнен оператором 2026-04-24):**

```bash
cd /Users/protey/pili-crm-data/tg-exports/
mkdir "+77471057849"
mv DataExport_2026-04-11 "+77471057849/"
mkdir "+79161879839"
# вторая выгрузка кладётся в +79161879839/ плоско (без промежуточной папки с датой)
```

### 6. Изменения в `ingestion/tg_import.py`

**Автодетект аккаунта из структуры папок.** Основной сценарий — оператор запускает `python3 -m ingestion.tg_import` без параметров, скрипт сам определяет, какую выгрузку импортировать.

**Логика `--input-dir`:**

| Что передано | Что делает скрипт |
|---|---|
| Ничего (`--input-dir` не указан) | Сканирует `/Users/protey/pili-crm-data/tg-exports/`, находит подпапки с именами в E.164 формате, для каждого аккаунта находит `result.json` (сначала прямо в `+PHONE/`, потом — в самой свежей `+PHONE/DataExport_*/`). Если аккаунтов несколько — требует явного `--account-phone` или `--input-dir`. Если один — импортирует его. |
| Папка аккаунта (`+77471057849/`) | Ищет `result.json`: приоритет — плоская структура (прямо в папке аккаунта); fallback — самая свежая `DataExport_*/` внутри. Аккаунт — по имени папки. |
| Папка выгрузки legacy (`.../+77471057849/DataExport_2026-04-11/`) | Импортирует указанную выгрузку. Аккаунт — по имени родительской папки аккаунта (должно совпадать с E.164 regexp). |
| Плоская структура (`.../+79161879839/` с `result.json` прямо внутри) | Импортирует прямо эту выгрузку. Аккаунт — по имени папки. |
| Устаревший плоский путь **без папки-аккаунта** (`.../DataExport_2026-04-11/` вне `+PHONE/` обёртки) | Отказ с явной ошибкой: «Папка выгрузки должна быть внутри папки-аккаунта с именем E.164. См. ADR-012, раздел 5.» |

**Приоритет поиска `result.json` внутри `+PHONE/`:**

1. `+PHONE/result.json` (плоская структура — новый стандарт)
2. `+PHONE/DataExport_YYYY-MM-DD/result.json` (legacy, при наличии нескольких — самая свежая по дате в имени)

Если оба варианта присутствуют одновременно — **плоская структура имеет приоритет**. Это защита от случая, когда legacy-папка осталась как артефакт после миграции на новый формат.

**Опциональный `--account-phone`:** при указании **переопределяет** автодетект, **но сверяется** с именем папки. При несовпадении — ошибка. Назначение: подстраховка, если оператор импортирует что-то из нестандартного места.

**Проверка JSON:** после определения аккаунта скрипт сверяет `phone_number` с `personal_information.phone_number` из `result.json`. При несовпадении — ошибка. Это защита от перепутанных папок.

**Поведение скрипта:**

1. Определить phone_number аккаунта (из `--input-dir` / дефолтной папки / `--account-phone`)
2. Верифицировать сверкой с JSON
3. `SELECT * FROM communications_telegram_account WHERE phone_number = ?`. Если запись отсутствует — отказ с ясной ошибкой: «Аккаунт не зарегистрирован. Добавьте через `python3 -m ingestion.register_account --phone +XXX --display-name 'Страна (+XXX)'`.»
4. Если `telegram_user_id` у аккаунта NULL — записать из `personal_information.user_id` (автодетект)
5. Все `INSERT` в `communications_telegram_chat` выполнять с `owner_account_id = <найденный>`
6. `ON CONFLICT (owner_account_id, telegram_chat_id) DO NOTHING` — новая пара для конфликт-резолюции
7. Обновить `last_import_at = now()` у аккаунта. Если `first_import_at` IS NULL — также заполнить.

**Множественные выгрузки одного аккаунта.** Сценарий: через несколько месяцев оператор делает новую выгрузку казахстанского аккаунта для обновления данных. При плоской структуре (новый стандарт) новая выгрузка **заменяет** содержимое `+77471057849/` (см. раздел 10 «Политика хранения выгрузок»). При запуске `tg_import` скрипт увидит только актуальную выгрузку — что правильно, старая информация уже в БД через предыдущий импорт. `ON CONFLICT DO NOTHING` обеспечит, что уже импортированные сообщения не продублируются, а новые (которых не было в предыдущей выгрузке) — добавятся. Политика: **замена выгрузки на диске + полагаться на `ON CONFLICT` для дедупликации на уровне БД**. Не требуется хранить watermark «до какой даты уже импортировано» — конфликт-резолюция на уровне БД гарантирует корректность.

**Важное свойство Telegram Desktop.** Функции выбрать период выгрузки нет — каждая выгрузка содержит **всю историю чата от начала времён**. При повторной выгрузке того же аккаунта `ON CONFLICT DO NOTHING` пропустит ~99% уже существующих сообщений, допишет только новые. Это делает стратегию устойчивой без необходимости явных watermarks в коде. Следствие — новый импорт не подхватит правки уже существующих сообщений (если бы они были отредактированы в Telegram после первого импорта): запись уже есть, `ON CONFLICT` её не трогает. Для MVP это не проблема — подробнее см. открытый вопрос в `06_open_questions.md`.

### 7. Новый скрипт `ingestion/register_account.py`

Минимальный CLI для регистрации аккаунта перед первым импортом его выгрузки.

Параметры:

| Флаг | Обязательность | Назначение |
|---|---|---|
| `--phone PHONE` | required | Номер в E.164 |
| `--display-name NAME` | required | Человекочитаемая метка |
| `--notes NOTES` | optional | Свободные заметки |
| `--telegram-user-id ID` | optional | Собственный user_id в этом аккаунте. Автодетект из первой JSON-выгрузки, если не указано. |

Поведение: INSERT в `communications_telegram_account`, при конфликте на phone_number — сообщение «аккаунт уже зарегистрирован» и exit code 1 (не 0 — чтобы в скриптах легко отличить первую регистрацию от повторной).

Пример для второго аккаунта:

```bash
python3 -m ingestion.register_account \
  --phone +79161879839 \
  --display-name "Россия (+79161879839)"
```

### 8. Изменения в MCP-tool `link_chat_to_customer`

Минимально: tool **продолжает работать как есть**, потому что связь «chat → customer» через `communications_telegram_chat.customer_id` не меняется. Один клиент может быть привязан к нескольким чатам.

Добавить информационную проверку:

- При режиме `link_to_existing` или `create_new` — перед коммитом проверить, нет ли уже другого чата с этим же `customer_id` (если linking) или с этим же `telegram_id` на сообщающем пользователе (если `create_new`). Если есть — добавить в ответ tool'а поле `existing_channels: list[dict]` с записями `{chat_id, owner_account_display_name, message_count}`.
- Cowork при получении этого поля показывает оператору: «Этот клиент уже привязан к N каналам: [список]. Это ожидаемо (клиент пишет с двух аккаунтов), убедитесь, что это тот же человек.»

Это НЕ блокирует привязку. Это информационный сигнал.

### 9. Публичный интерфейс модуля `communications`

Новые функции в `app/communications/service.py`:

- `get_account_by_phone(phone: str) -> TelegramAccount | None`

### 10. Политика хранения выгрузок на диске

**На диске одновременно хранится одна актуальная выгрузка на аккаунт.** Исторические версии не накапливаются.

Обоснование:

- **БД — источник правды.** После успешного импорта все содержательные данные (чаты, сообщения, метаданные) находятся в PostgreSQL. JSON-файлы после импорта не используются системой — они нужны только как исходное сырьё для импорта.
- **Размер выгрузки значителен.** Типовой размер ~6 ГБ на аккаунт. Накопление 10 версий × 2 аккаунта = 120 ГБ ради данных, которых уже нет ничего, чего нет в БД.
- **Inflight защита.** Последняя выгрузка **сохраняется** до появления следующей — на случай, если импорт оказался некорректным и нужно перезапустить. Удаляется **только при замене** новой выгрузкой, не сразу после импорта.

**Рабочий процесс замены выгрузки:**

```bash
# Новая выгрузка положена, например, в ~/Downloads/Telegram_Export_2026-09-15/
# Заменяем старую для казахстанского аккаунта:

cd /Users/protey/pili-crm-data/tg-exports/+77471057849/
rm -rf chats/ profile_pictures/ result.json DataExport_*/   # снос старого
cp -r ~/Downloads/Telegram_Export_2026-09-15/* ./            # копирование нового

# Далее импорт работает штатно:
cd /Users/protey/pili-crm
python3 -m ingestion.tg_import --input-dir /Users/protey/pili-crm-data/tg-exports/+77471057849/
```

Этот процесс — документирован в `ingestion/README.md`, автоматизация через скрипт не требуется (ручной контроль при обновлении снижает риск случайной потери).

**Первая выгрузка казахстанского аккаунта (legacy).** `+77471057849/DataExport_2026-04-11/` существует как legacy-структура. Скрипт её поддерживает. При следующей выгрузке казахстанского аккаунта (планируется) — переходим на плоскую структуру: содержимое новой выгрузки ложится прямо в `+77471057849/`, legacy-папка удаляется в рамках того же шага замены.

## Последствия

### Что становится возможным

- Корректный импорт второй выгрузки без схлопывания чатов с совпадающими `telegram_chat_id` между аккаунтами
- Один физический клиент — несколько Telegram-чатов (по одному на каждый аккаунт), все связаны с одной карточкой `orders_customer`
- Отчётность и аналитика по каждому аккаунту отдельно
- Оператор запускает `python3 -m ingestion.tg_import` без параметров — скрипт сам разберётся, что импортировать
- Структура папок на диске говорит сама за себя — через 6 месяцев очевидно, какая выгрузка к какому аккаунту относится

### Какие ограничения появляются

- **Миграция существующих данных:** 303 чата backfill-ятся как принадлежащие казахстанскому аккаунту. Если где-то в реальности (в `tool-shop-crm/` или в оперативной работе оператора) есть данные, созданные в предположении «все чаты одного аккаунта» — при добавлении российского аккаунта эти предположения нужно пересмотреть.
- **Перенос папок на диске:** разовый операционный шаг, выполнен оператором 2026-04-24.
- **Дубликаты клиентов** — рискованная зона. Клиент Иван написал на казахстанский (telegram_id=111, chat_id=100), оператор привязал к клиенту Ivan id=500. Потом Иван написал на российский (telegram_id=111, chat_id=200). В `link_chat_to_customer` оператор должен увидеть, что telegram_id=111 уже привязан к id=500, и привязать новый chat к тому же клиенту, не создавая дубля. Это уже покрыто существующим механизмом `telegram_id_conflict` (ADR-010 Задание 2), плюс новое поле `existing_channels` делает сигнал явным.

### Операционные заметки

- **Preferred channel.** У клиента может быть любимый канал — тот, через который он чаще пишет. Сейчас это не хранится явно, оператор видит это через частоту сообщений. В будущем — возможное расширение `orders_customer` полем `preferred_telegram_account_id`, но для MVP не обязательно.
- **Консолидация.** Если клиент засветился в двух аккаунтах — это нормально, но по возможности оператор должен стимулировать его писать в один. Это операционное правило, не техническое. В системе никаких автоматических уведомлений и блокировок.

### Что придётся учитывать дальше

- ADR-010 Задание 3 (Telethon incremental) — при реализации должен использовать тот же справочник аккаунтов. Каждый аккаунт — своя API-сессия Telethon, свой инкрементальный polling, свои credentials.
- ADR-011 (анализатор) — работает по `chat_id`, не по аккаунту. Один клиент с двумя чатами — два отдельных анализа, два narrative в `analysis_chat_analysis`. Это ок на MVP, но в будущем может появиться отдельный мета-анализ «все каналы клиента вместе».
- Google Sheets-зеркало (ADR-005) — при экспорте `communications_telegram_chat` стоит добавить колонку `owner_account_display_name` для читаемости.
