# ADR-006: Размещение derivation rule (derive-status) на уровне PostgreSQL

**Статус:** принят
**Дата:** 2026-04-22
**Связанные ADR:** ADR-001 v2 (модульный монолит), ADR-002 (стек), ADR-003 final-ready (схема), ADR-003 Addendum (статусы, derivation rule)
**Связанные документы:** `docs/cowork-system-prompt.md` (раздел 5 — статусы), `crm-mcp/IMPROVEMENTS.md` (запись I-1)

---

## Контекст

ADR-003 Addendum зафиксировал **derivation rule**: статус заказа (`orders_order.status`) автоматически вычисляется по статусам позиций (`orders_order_item.status`) по правилу «item-level min → order-level». Это — инвариант данных: статус заказа есть функция от статусов позиций.

Текущая реализация: derivation rule живёт в FastAPI как endpoint `derive-status`, который вызывается MCP-сервером после каждого `update_order_item_status`. Это создаёт проблему зависимости инварианта данных от доступности прикладного сервера.

**Конкретный инцидент (IMPROVEMENTS.md, запись I-1):** позиция в заказе З-271 переведена в статус `ordered`, но статус заказа не изменился — FastAPI был недоступен. Потребовался ручной вызов `derive-status` после перезапуска. Данные находились в несогласованном состоянии в течение времени простоя.

Это не эргономическая проблема. Это **архитектурный дефект**: инвариант данных не должен зависеть от доступности application server. Требуется ADR, фиксирующий правильное место для derive-status логики.

Решение принимается сейчас, потому что:
- инцидент уже произошёл и задокументирован;
- последствия рассогласования серьёзные (оператор видит неверный статус заказа, принимает на его основании решения);
- чем дольше логика остаётся в FastAPI, тем больше кода завязывается на текущую схему;
- цена переноса сейчас — одна миграция; цена переноса через полгода — разбор зависимостей во всех новых write-путях.

## Варианты

### Вариант A. PostgreSQL-триггер AFTER UPDATE на `orders_order_item`

Логика derivation rule реализуется как функция PL/pgSQL + триггер `AFTER UPDATE OF status` на таблице `orders_order_item`. При любом изменении статуса позиции триггер атомарно пересчитывает статус соответствующего заказа в той же транзакции.

- **Плюсы:**
  - Атомарность: обновление item.status и order.status в одной транзакции, невозможно оставить несогласованное состояние.
  - Независимость: работает при прямом SQL-доступе, из миграции, из любого tool, включая ещё не написанные.
  - Одно место истины для правила.
  - Минимум компонентов: одна функция, один триггер.
- **Минусы:**
  - Бизнес-логика частично уходит в БД (не все разработчики в Python-команде свободно читают PL/pgSQL).
  - Дебаг сложнее (нужен `RAISE NOTICE`, настроенный клиент).
  - Side-effects (уведомления, логирование в прикладном коде) из триггера выполнить нельзя — только изменения данных.

### Вариант B. Логика в MCP-слое (Python)

Функция derive-status импортируется в MCP-сервер и вызывается явно внутри `update_order_item_status` в той же транзакции. FastAPI из цепочки исключается.

- **Плюсы:**
  - Python-код, привычный стек.
  - Лёгкое тестирование (unit-тесты как у pricing).
  - Side-effects размещаются здесь же.
  - Транзакционность через SQLAlchemy session.
- **Минусы:**
  - Дублирование при появлении новых путей изменения item.status (bulk-операция, админка, миграция) — функцию нужно звать из всех мест.
  - Прямой UPDATE в psql не триггерит пересчёт.
  - Риск забыть вызов при добавлении нового write-tool.

### Вариант C. Остаётся в FastAPI + компенсирующий механизм

Retry при недоступности FastAPI в MCP, healthcheck, периодический reconciliation job, находящий и исправляющий рассогласования.

- **Плюсы:**
  - Минимум изменений в текущей реализации.
  - Бизнес-логика остаётся в прикладном слое.
- **Минусы:**
  - Три новых механизма (retry, очередь, reconciliation), каждый со своими edge cases.
  - Окно рассогласования не устраняется, а сдвигается во времени.
  - Архитектурно не решает проблему: консистентность зависит от доступности application server.

### Вариант D (гибрид). Триггер + сервисный слой side-effects

Derivation rule — в PostgreSQL-триггере (инвариант). Side-effects (уведомления, логирование в прикладных таблицах) — в Python-коде MCP/FastAPI поверх результата триггера, читается через RETURNING или повторный SELECT.

- **Плюсы:** консистентность гарантируется БД; side-effects остаются в Python; явное разделение «данные vs действия».
- **Минусы:** больше компонентов, чем в чистом варианте A; требует аккуратного ловли изменения order.status в Python-слое.

## Критерии выбора

- **Консистентность данных (инвариант).** Ключевой критерий. Derivation rule — не «логика», а правило целостности. Инвариант не должен нарушаться при любых путях изменения.
- **Независимость от доступности application server.**
- **Простота (минимум движущихся частей).**
- **Дебажность.**
- **Testability.**
- **Совместимость с текущим стеком (ADR-002: PostgreSQL 16, Alembic для миграций).**

---

## Принятое решение

Принят **Вариант A — PostgreSQL-триггер AFTER UPDATE на `orders_order_item`**.

### Ключевые параметры

1. **Функция** `derive_order_status(order_id BIGINT) RETURNS orders_order_status` — детерминированная, stable, принимает id заказа, возвращает вычисленный статус по правилу из ADR-003 Addendum.

2. **Триггерная функция** `trg_order_item_derive_status()` — вызывается триггером, читает NEW.order_id, вызывает `derive_order_status`, обновляет `orders_order.status` если изменился.

3. **Триггер** `orders_order_item_derive_status_trg` — `AFTER UPDATE OF status ON orders_order_item FOR EACH ROW`. Срабатывает только при изменении поля `status` (не при других UPDATE).

4. **Дополнительный триггер** `AFTER INSERT OR DELETE ON orders_order_item` — для случаев добавления/удаления позиции (редкие, но возможные в seed и миграциях данных).

5. **Идемпотентность:** функция проверяет, изменился ли вычисленный статус, и обновляет `orders_order.status` только при изменении. Предотвращает рекурсию и лишние WAL-записи.

6. **Логика derive-status остаётся единой с ADR-003 Addendum.** SQL-реализация — это **перенос существующего Python-кода**, а не новое правило. Правило не меняется.

7. **Endpoint `derive-status` в FastAPI** удаляется после подтверждения работоспособности триггера. MCP перестаёт его вызывать.

8. **MCP-tool `update_order_item_status` упрощается:** больше не делает HTTP-вызов в FastAPI, просто выполняет UPDATE item.status — триггер срабатывает автоматически. MCP может вернуть новый статус заказа, сделав один дополнительный SELECT `orders_order.status` после UPDATE (в той же транзакции, после триггера это уже новое значение).

9. **Side-effects** (уведомления, логирование смены статуса в прикладных таблицах) в этом ADR **не предусматриваются**. Когда они появятся — вводится вариант D (триггер + Python-слой на RETURNING), отдельным аддендумом к этому ADR.

10. **Зависимости между модулями:** триггер живёт в модуле `orders` (принадлежит таблицам `orders_order_item` / `orders_order`). Не создаёт кросс-модульных зависимостей. Соответствует ADR-001 v2.

## Последствия

### Что становится проще

- Невозможно оставить рассогласованное состояние item.status ↔ order.status при любом пути изменения.
- MCP-tool `update_order_item_status` упрощается: нет HTTP-вызова, нет retry-логики, нет обработки недоступности FastAPI.
- Снимается зависимость инварианта данных от FastAPI.
- Любой новый write-путь к order_item.status автоматически защищён триггером — не нужно помнить про derive-status.
- Закрывается запись I-1 в `IMPROVEMENTS.md`.

### Какие ограничения появляются

- Часть бизнес-логики живёт в PL/pgSQL — требует минимальной компетенции в команде.
- Дебаг триггера сложнее обычного Python-кода; нужна дисциплина логирования через `RAISE NOTICE` при разборе инцидентов.
- Side-effects по смене статуса нельзя вешать напрямую на триггер — когда понадобятся, нужен аддендум на вариант D.
- Изменение правила derivation — через миграцию, не через правку Python.
- Bulk-UPDATE большого числа позиций будет медленнее, чем без триггера (N вызовов функции). Для текущего объёма (десятки-сотни заказов в период) — не проблема.

### Что придётся учитывать дальше

- При добавлении новых статусов в enum `orders_order_status` или `orders_order_item_status` — обновлять функцию `derive_order_status` через миграцию.
- При появлении требования side-effects (уведомления, аудит) — написать аддендум на вариант D.
- При рефакторинге таблицы `orders_order_item` (переименование поля status, разделение) — миграция должна также обновлять функцию и триггер.
- Тесты триггера писать на Python через pytest + реальный PostgreSQL (инфраструктура уже есть по ADR-002).

## Что должен сделать Claude Code

Передаётся в Prompt Factory for Claude Code отдельными промтами:

1. **Миграция Alembic** на создание:
   - функции `derive_order_status(BIGINT) RETURNS orders_order_status` на PL/pgSQL, реализующей правило ADR-003 Addendum (учёт отменённых позиций — отменённые игнорируются; если все позиции отменены — статус заказа `cancelled`; иначе — минимум по неотменённым);
   - триггерной функции `trg_order_item_derive_status()`;
   - триггеров `AFTER UPDATE OF status` и `AFTER INSERT OR DELETE` на `orders_order_item`;
   - downgrade-секции, удаляющей всё созданное.

2. **Упрощение MCP-tool** `update_order_item_status`:
   - убрать HTTP-вызов `derive-status` в FastAPI;
   - после UPDATE выполнить SELECT нового `orders_order.status` в той же сессии и вернуть вместе с item.status;
   - при изменении order.status по derivation — показать оператору новый статус заказа (русская метка).

3. **Удаление FastAPI-endpoint** `/derive-status` после подтверждения работоспособности триггера. До подтверждения — оставить как fallback, пометить `@deprecated`.

4. **Тесты (pytest + реальный PostgreSQL):**
   - перевод единственной позиции заказа в `ordered` → статус заказа `in_procurement`;
   - все позиции `arrived`, одна `delivered` → статус заказа `arrived`;
   - все позиции `delivered` → статус заказа `delivered`;
   - все позиции `cancelled` → статус заказа `cancelled`;
   - одна `cancelled`, одна `ordered` → статус заказа `in_procurement` (отменённая игнорируется);
   - INSERT новой позиции в существующий заказ с позициями `delivered` → статус заказа пересчитывается;
   - идемпотентность: повторный UPDATE с тем же значением status не приводит к рекурсии и лишним записям;
   - проверка работы триггера при недоступности FastAPI (физический тест — остановить FastAPI, сделать UPDATE через MCP, проверить что order.status обновлён).

5. **Обновление `crm-mcp/IMPROVEMENTS.md`:**
   - запись I-1 → статус `done` с связанным решением `ADR-006`.
