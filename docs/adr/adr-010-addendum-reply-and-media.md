## Addendum [2026-04-23] — reply и медиа-метаданные как first-class поля

**Контекст.** При разведке фактической структуры Telegram Desktop JSON Export (303 чата, 32 МБ `result.json`, все `personal_chat`) стало ясно, что два аспекта, которые в основной версии ADR-010 обрабатывались минималистично, заслуживают более строгого подхода:

1. **Reply-сообщения** — оператор часто использует reply для связи короткого ответа («отдал вот этот», «да, 8000 руб») с сообщением-контекстом (перечень товаров, фото детали). Без связи reply текст теряет смысл.
2. **Медиа-метаданные** — сообщения с медиа часто содержат ценный текст в подписи. Метаданные нужны не только для факта «тут было фото», но и для возможности открыть его из локальной папки при разборе чата.

**Решение.**

### 1. `reply_to_telegram_message_id` — отдельная колонка

В таблице `communications_telegram_message` добавляется колонка:

| Поле | Тип | Назначение |
|---|---|---|
| `reply_to_telegram_message_id` | `TEXT NULL` | Telegram message_id сообщения, на которое ссылается reply. NULL = обычное сообщение, не reply. |

**FK на самого себя не ставим.** Исходное сообщение может быть удалено в Telegram до экспорта, не попасть в выгрузку или быть импортировано позже — FK в любом из этих случаев нарушится. Связь разрешается опционально при чтении: `SELECT ... WHERE chat_id = :chat AND telegram_message_id = :reply_to`.

**Partial composite index:**

```sql
CREATE INDEX ix_telegram_message_reply_to
ON communications_telegram_message (chat_id, reply_to_telegram_message_id)
WHERE reply_to_telegram_message_id IS NOT NULL
```

Индекс покрывает только сообщения с reply — меньшинство. Поиск оригинала при разборе чата — O(log n).

**Дополнительно:** `reply_to_message_id` из исходного JSON также сохраняется целиком в `raw_payload` — для возможной форензики и на случай, если потребуются другие поля reply-метаданных.

### 2. Медиа-метаданные в `raw_payload` — расширенный состав

В `raw_payload JSONB` сообщений с медиа сохраняется **полный блок**, достаточный для последующего открытия файла из локальной подпапки чата:

```json
{
  "media_type": "photo",
  "file_name": "photo_2024-05-01_123456.jpg",
  "relative_path": "chats/chat_180/photos/photo_2024-05-01_123456.jpg",
  "file_size_bytes": 245120,
  "mime_type": "image/jpeg",
  "original_message_fields": { ... }
}
```

`relative_path` — относительно корня JSON-экспорта. Абсолютный путь строится как `{TELEGRAM_EXPORT_ROOT}/{relative_path}` на стороне читателя.

Само физическое копирование медиа в `~/pili-crm-media/telegram/` остаётся за разбором чата оператором (фаза 3 ADR-010, правило ADR-010 «оператор явно подтверждает нужность»). Ingestion только фиксирует метаданные.

### 3. Сообщения с медиа и непустым текстом

Сообщения, у которых есть и `media_type`, и непустой `text`, **импортируются**: текст — в поле `text`, метаданные медиа — в `raw_payload` по схеме выше. Эти сообщения видны в переписке и разбираются наравне с текстовыми.

### 4. Уточнённое правило фильтрации сообщений

| Тип | Импорт |
|---|---|
| Только текст (`text != ''`, нет `media_type`) | ✅ |
| Текст + media_type `photo` / `file` / `video` | ✅ (текст + метаданные) |
| Только media_type, без текста (`text == ''`) | ✅ только метаданные в `raw_payload`, `text = NULL` |
| `media_type = voice_message` или `sticker`, без текста | ❌ пропускается |
| `type = 'service'` | ❌ пропускается |

**Последствия.** Reply-сообщения становятся first-class полем, поиск оригинала при чтении чата проходит по индексу. Медиа-метаданные всегда сохранены в достаточной полноте. Миграция — одна колонка + partial index, реализуется отдельным мини-пакетом до Задания 1 ADR-010 (`tg_import.py`).

**Связанные вопросы (в `06_open_questions.md`):** policy автоматического копирования медиа-файлов при подтверждении оператором — не входит в Задание 1, решается при реализации MCP-tool `link_chat_to_customer` (Задание 2).
