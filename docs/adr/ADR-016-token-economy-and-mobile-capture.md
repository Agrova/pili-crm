# ADR-016: Экономия токенов на рутине + мобильный capture-канал

**Статус:** принят
**Дата:** 2026-04-30
**Связанные ADR:** ADR-001 v2 (модульный монолит, Cowork+MCP как интерфейс), ADR-005 (mirror Google Sheets), ADR-010 (Telegram ingestion pipeline)
**Связанные документы:** `docs/cowork-system-prompt.md`, `crm-mcp/IMPROVEMENTS.md`, `docs/PLAN.md` (группа G18, G14)

---

## Контекст

ADR-001 v2 фиксирует Claude Cowork + MCP-сервер как единственный пользовательский интерфейс CRM. Эта модель оптимальна для задач, требующих рассуждений (разбор identity quarantine, link chats, сборка заказа из переписки), но создаёт две операционные боли по мере накопления опыта работы:

**Боль 1 — рутина без рассуждений идёт через токены LLM.**
Простые операторские задачи — посмотреть pending-заказы, найти клиента по фрагменту имени, переключить статус позиции `pending → ordered`, посмотреть курс — сейчас выполняются через диалог с Cowork. Каждая такая операция тратит токены на чтение MCP-ответа, форматирование таблицы, иногда подтверждение. Оператор интуитивно ощущает: «то что не требует рассуждений должно дублироваться интерфейсом не требующим токенов». Конкретного измерения боли нет — это интуитивное желание сэкономить.

**Боль 2 — невозможно зафиксировать момент в дороге.**
Оператор часто на улице или на встречах с клиентами. Единственный канал — телефон (Android, опционально iPhone). Сейчас, чтобы зафиксировать «клиент Воропаев попросил уточнить срок З-1580» или «нужно завтра позвонить про доставку», оператор делает заметку в постороннем приложении, потом вручную переносит её в Cowork на Mac. Половина моментов теряется или забывается.

Что **не обсуждается** в этом ADR: полный мобильный CRM-клиент (G14 — отдельная задача в спящем режиме); полная BI-аналитика; ролевая система (один оператор, без auth по ADR-001 v2).

Что важно учесть как ограничение:
- **Разумная достаточность** — один пользователь, малый магазин. Нельзя поднимать сложную инфру (домен, HTTPS, Cloudflare Workers, Kubernetes).
- **Cowork+MCP остаётся главным интерфейсом** — новые каналы дополняют, не заменяют его.
- **Безопасность** — write-операции остаются за Cowork на Mac, мобильный канал read+capture без write в основные таблицы.

## Варианты

Решение распадается на **две независимые развилки**, потому что desktop-оптимизация и mobile-capture — разные задачи с разными ограничениями. Они принимаются вместе, чтобы не плодить пересекающихся ADR.

### Развилка A — Desktop UI для рутины без рассуждений (на Mac)

#### Вариант A1 — Cowork live artifacts (HTML-виджеты на стороне Cowork) ✅

Cowork предоставляет API `mcp__cowork__create_artifact` — self-contained HTML-страница, которая persistent между сессиями и пулит данные через `window.cowork.callMcpTool(name, args)` напрямую от MCP-сервера. Чтение MCP-tool из артефакта **не тратит токены LLM** — это прямой вызов tool-а, ответ показывается на странице.

Артефакты, которые покрывают типовую рутину:
- **«Pending заказы»** — таблица с фильтрами по статусу/клиенту, кнопка Reload, переход в Cowork по нужной строке.
- **«Карточка клиента»** — поиск по имени/телефону/Telegram, история заказов, профиль. Без Cowork-диалога.
- **«Курс и pricing»** — текущий курс, последние операции обмена, быстрая прикидка цены.

Артефакты вызываются командой Cowork один раз («покажи дашборд pending») и потом живут самостоятельно — оператор открывает их закладкой.

- **Плюсы:** ноль внешней инфры; реиспользует существующий MCP; persistent через сессии; нативно встроено в Cowork; написание артефакта — один промт через Prompt Factory; write-операции остаются под контролем Cowork (артефакты read-only по умолчанию).
- **Минусы:** артефакты живут внутри Cowork-приложения (не открываются вне Mac); ограничения на JS-библиотеки (Chart.js / Grid.js / Mermaid из CDN, остальное inline).

#### Вариант A2 — Streamlit / FastAPI + Jinja-страница на localhost:8001

Поднять отдельное мини-веб-приложение, которое читает БД через FastAPI и рендерит дашборды. Открывается в браузере по `http://localhost:8001/dashboard`.

- **Плюсы:** независимо от Cowork; больше свободы в UI и библиотеках; те же страницы можно открыть с телефона при тоннеле (B-вариант).
- **Минусы:** требует разработки фронта (Streamlit или Jinja-templates); FastAPI уже есть, но новые роуты — отдельная работа; дублирует то, что Cowork artifacts даёт «из коробки».

#### Вариант A3 — Native Mac-приложение (SwiftUI / Tauri)

- **Плюсы:** нативный UX.
- **Минусы:** оверкилл для одного пользователя; требует поддержки бинарника, подписи, обновлений; нет выигрыша против браузера на той же машине.

#### Вариант A4 — Использовать Google Sheets mirror (ADR-005) как UI

- **Плюсы:** частично уже работает; знакомый интерфейс.
- **Минусы:** read-only; обновление раз в сутки (ADR-005); неинтерактивно — нет кнопок «открыть в Cowork»; не подходит для оперативной рутины.

### Развилка B — Mobile capture (зафиксировать момент в дороге)

#### Вариант B1 — Telegram-бот в режиме capture-only inbox ✅

Минимальный Telegram-бот с одной целью: принять capture от оператора и сложить в БД в новую таблицу `inbox_capture`. Не CRM-клиент, не write в основные таблицы — чёрный ящик заметок.

Команды:
- `/note <текст>` — общая заметка.
- `/lead <текст>` — потенциальный заказ/клиент (хайлайт для разбора).
- Любая фотография → capture с фото и caption.
- `/list` — последние N заметок (read-only справка для самопроверки).

Разбор на Mac:
- Новый MCP-tool `list_inbox_captures(status='pending', limit=20)` — Cowork показывает inbox.
- MCP-tool `mark_inbox_capture(capture_id, action='processed'|'archived')` — после переноса в основные таблицы.
- Перенос в основные таблицы — вручную через Cowork (`create_customer`, `create_order`, `update_order_item_status`), Cowork ссылается на capture для контекста.

Инфра:
- Бот регистрируется через `@BotFather` (бесплатно, без домена).
- Long-polling (Telethon) внутри FastAPI lifespan или отдельным процессом через launchd plist (как в G9).
- Доступ ограничен по `telegram_user_id` оператора — никаких других пользователей бот не обслуживает.

- **Плюсы:** минимум инфры (только бот-токен); работает с любого телефона (Android/iPhone); нет открытых портов; переиспользует Telethon уже в проекте; capture не нарушает write-инвариант (только в `inbox_capture`, не в основные таблицы); чётко отделён от полного G14.
- **Минусы:** UI ограничен возможностями Telegram (нет своего UI); бот должен быть запущен 24/7 на Mac (риск пропусков при сбое); нужна новая таблица + два MCP-tool + бот-сервис.

#### Вариант B2 — PWA через Cloudflare Tunnel / Tailscale

FastAPI отдаёт мобильную страницу, тоннель пробрасывает наружу.

- **Плюсы:** богатый UI (HTML/CSS); те же страницы, что A2.
- **Минусы:** требует домен или Cloudflare аккаунт; HTTPS/auth/tokens — нетривиальная инфра для одного пользователя; «разумная достаточность» нарушена.

#### Вариант B3 — Telegram «Saved Messages» + парсер

Оператор пишет себе в избранное, скрипт раз в час забирает через Telethon и складывает в `inbox_capture`.

- **Плюсы:** ноль новой инфры (Telethon уже есть); ничего не настраивать на телефоне.
- **Минусы:** парсинг произвольного текста (где заметка, где личное?); коллизия с обычным использованием Saved Messages; нет команд `/lead`, `/note` — нет структуры.

#### Вариант B4 — Native Android-приложение

- **Минусы:** оверкилл; разработка под Android; iPhone-сценарий не покрыт.

## Критерии выбора

- **Разумная достаточность инфры:** A1 — ноль; B1 — только бот-токен. Это нижняя граница для выбранного scope.
- **Совместимость с ADR-001 v2:** A1 встроен в Cowork — это естественное расширение текущего интерфейса. B1 — отдельный канал, но read+capture only, write остаётся за Cowork.
- **Минимизация работы под неподтверждённую боль:** A1 — артефакт делается одним промтом через Prompt Factory, легко выкинуть если не зайдёт. B1 — базовый бот тоже компактен (одна таблица + два MCP-tool + 200 строк бота).
- **Безопасность данных:** B1 ограничен только capture в чёрный ящик — даже если бот будет скомпрометирован, основные таблицы остаются нетронутыми. A1 read-only по умолчанию.
- **Возможность отката:** A1 — артефакты выкидываются командой; B1 — бот глушится, таблица `inbox_capture` остаётся как архив заметок.

## Принятое решение

### Двухчастное решение

**Часть 1 — Desktop оптимизация через Cowork live artifacts (вариант A1).**
**Часть 2 — Mobile capture-only Telegram-бот (вариант B1).**

Обе части реализуются в одной группе плана G18, но как **две независимые подзадачи** с разными промтами и потенциально разной приоритетностью реализации.

### Почему одно ADR, а не два

Правило проекта (`CLAUDE.md`, секция «Документооборот») — «один документ — одна задача». Здесь задача одна: **разделить операции CRM на «требуют рассуждений» (остаются в Cowork) и «не требуют» (выводятся в безтокенные интерфейсы)**. Desktop-артефакты и mobile-capture — две стороны одной границы:

- Desktop без рассуждений = «дашборд» — оператор смотрит то, что и так знает (pending, клиент, курс).
- Mobile capture = «inbox» — оператор фиксирует то, что разберётся позже в Cowork (то есть рассуждения тоже не на телефоне).

Если разделить на ADR-016a и ADR-016b, эта общая граница потеряется, и в каждом отдельно придётся повторять обоснование «почему не делаем write через бот / почему артефакт read-only». Поэтому одно решение, две подзадачи в G18.

### Жёсткое правило: бот = только capture

Telegram-бот из этого ADR **навсегда** ограничен `inbox_capture` и двумя MCP-tool разбора. Любое расширение бота на write в основные таблицы (`orders_*`, `catalog_*`, `warehouse_*`, `communications_*`, `analysis_*`, `pricing_*`, `finance_*`) — **нарушение архитектуры**, требующее нового ADR (G14, полный мобильный CRM, с пересмотром принципа «Cowork+MCP — единственный интерфейс write» из ADR-001 v2).

Это правило защищает три инварианта:
- Cowork остаётся единственной точкой write-операций (ADR-001 v2).
- Защита от компрометации бот-токена — даже атакующий с токеном не может портить основные данные.
- Понятная граница ответственности — всё, что нужно подтвердить или подумать, делается на Mac.

### Часть 1 — Desktop live artifacts

Создаются три стартовых артефакта, покрывающих типовую рутину без рассуждений:

| Артефакт | Tool-вызовы | Назначение |
|---|---|---|
| `pending_orders_dashboard` | `pending_orders` | Все pending заказы с фильтрами (статус, клиент, дата). Кнопка «открыть в Cowork» по строке. |
| `customer_lookup` | `find_customer`, `list_customers` | Поиск клиента по имени/телефону/Telegram, история заказов, Telegram deep link. |
| `pricing_reference` | (требует G17 — `get_current_exchange_rate`) | Текущий курс, последние обмены, быстрый калькулятор «$X → ₽Y». |

**Правила реализации артефактов:**
- Каждый артефакт = один HTML-файл, передаётся через `mcp__cowork__create_artifact` или хранится в `crm-mcp/artifacts/` для версионирования.
- Только `window.cowork.callMcpTool` — никакого прямого SQL.
- Только read-tools — write остаётся в Cowork-диалоге.
- `localStorage` для пользовательских фильтров (Cowork persistence).
- Все статусы — русские метки (правило из `cowork-system-prompt.md`).
- Каждый артефакт линкует строки на Cowork-команды через `sendPrompt(text)` — щелчок по pending-заказу шлёт в Cowork «покажи детали заказа №X».

**Что Cowork делает с артефактами:**
- При первой команде «покажи pending» — Cowork генерирует/обновляет артефакт через MCP.
- При повторных командах — Cowork отвечает «уже открыт в дашборде, обновить?».
- Сложные запросы («покажи pending Иванова за апрель с группировкой по поставщику») — Cowork отвечает диалогом, потому что это уже «требует рассуждений».

### Часть 2 — Mobile capture-only Telegram-бот

**Новая таблица `inbox_capture`:**

| Поле | Тип | Смысл |
|---|---|---|
| `id` | `BIGSERIAL PK` | |
| `captured_at` | `TIMESTAMPTZ NOT NULL DEFAULT NOW()` | Момент capture |
| `source` | `VARCHAR(20) NOT NULL` | `telegram_bot` (на старте — единственный) |
| `kind` | `VARCHAR(20) NOT NULL` | `note` / `lead` / `photo` |
| `text` | `TEXT NULL` | Содержание заметки или caption фотографии |
| `media_path` | `TEXT NULL` | Путь к фото в `media/inbox/` (если есть) |
| `status` | `VARCHAR(20) NOT NULL DEFAULT 'pending'` | `pending` / `processed` / `archived` |
| `processed_at` | `TIMESTAMPTZ NULL` | Момент разбора через Cowork |
| `processed_by_action` | `VARCHAR(40) NULL` | Что было сделано: `created_customer:123`, `created_order:Z-1580`, `archived` |
| `raw_metadata` | `JSONB NULL` | Telegram update raw для аудита |

**Команды бота:**

| Команда | Поведение |
|---|---|
| `/start` | Приветствие; бот проверяет `telegram_user_id` против разрешённого списка (один оператор) |
| `/note <текст>` | INSERT в `inbox_capture(kind='note', text=...)`; ответ «✓ записано как note #ID» |
| `/lead <текст>` | INSERT с `kind='lead'`; ответ «✓ записано как lead #ID» |
| `/list` | Последние 10 captures со статусом pending — read-only справка |
| `/help` | Список команд |
| _Фотография_ | Скачивается в `media/inbox/`, INSERT с `kind='photo'`, caption → `text` |
| _Любой другой текст_ | Трактуется как `/note <весь текст>` |

**Разрешённые пользователи:**
- Whitelist в env: `INBOX_BOT_ALLOWED_USER_IDS=12345678` (telegram_user_id оператора).
- Любой другой пользователь получает «доступ запрещён», update игнорируется.

**Два новых MCP-tool:**

- `list_inbox_captures(status='pending', limit=20, kind=None)` — read. Возвращает captures отсортированные по `captured_at DESC`.
- `mark_inbox_capture(capture_id, action, note=None)` — write. `action` ∈ {`processed`, `archived`}. `note` опционально записывается в `processed_by_action`.

**Сервис бота:**
- Файл `ingestion/inbox_bot.py` (Telethon-based, аналогично паттерну G9 `tg_incremental.py`).
- Запуск через launchd plist `~/Library/LaunchAgents/com.pilistrogai.inbox-bot.plist` (тот же паттерн что и FastAPI autostart G8).
- Логи в `logs/inbox-bot-*.log`.

**Workflow на Mac:**
1. Утром оператор спрашивает Cowork: «что в inbox?»
2. Cowork: `list_inbox_captures(status='pending')` → русский список с captures.
3. По каждой записи оператор решает: «создай клиента», «оставь как есть», «архивируй» — Cowork делает соответствующие операции, в конце `mark_inbox_capture(capture_id, action='processed', note='created_customer:NNN')`.

### Разделение токеноёмкости

| Тип задачи | Где исполняется | Токены LLM |
|---|---|---|
| Просмотр pending заказов | Артефакт A1 | Ноль |
| Поиск клиента по фрагменту имени | Артефакт A1 | Ноль |
| Просмотр текущего курса | Артефакт A1 | Ноль |
| Capture заметки в дороге | Telegram-бот B1 | Ноль (нет LLM) |
| Разбор inbox утром (рутина) | Cowork с использованием list_inbox_captures | Минимум — есть структурированные tools |
| Identity quarantine apply, link chats, сборка заказа из переписки | Cowork-диалог | Полные — здесь рассуждения нужны |

Это и есть граница «требует рассуждений / не требует рассуждений», которую оператор интуитивно ощущает.

## Последствия

### Что становится проще

- Оператор может посмотреть pending-заказы или найти клиента, не открывая Cowork-диалог — артефакт обновляется по кнопке Reload.
- Capture в дороге фиксируется в одном надёжном месте (БД), а не теряется в постороннем приложении.
- `inbox_capture` становится аудируемым архивом всех «надо не забыть» — даже если оператор не разобрал их сразу, ничего не пропадает.
- Telegram-бот переиспользует уже существующую Telethon-инфраструктуру (ADR-010), не вводит новой технологии.

### Какие ограничения появляются

- Артефакты живут внутри Cowork-приложения — без Cowork их не открыть. Это сознательное ограничение: write-операции должны идти через Cowork.
- Бот должен быть запущен 24/7 на Mac. При сбое Mac (как тепловой инцидент 2026-04-30) capture не доходят. Telegram сам не буферизует команды бота длительно — это известное ограничение, не блокер на стартовом scope.
- Артефакты могут устаревать при изменении MCP-tool сигнатур — придётся перегенерировать. Это нормальная плата за «один промт через Prompt Factory».
- Whitelist по `telegram_user_id` — единственная защита бота. Token бота должен храниться в `.env`, не коммититься.
- Появляется новая таблица `inbox_capture` — учесть в ADR-005 mirror (нужна ли вкладка для inbox).

### Безопасность бот-токена

- Бот-токен от @BotFather хранится **только** в `.env` (или `.env.local`), который **должен быть в `.gitignore`**.
- При утечке токена — немедленно вызвать `/revoke` у @BotFather, получить новый, обновить `.env`, перезапустить launchd.
- Whitelist `INBOX_BOT_ALLOWED_USER_IDS` — основная защита: даже владелец нового токена не сможет писать в `inbox_capture` без правильного `telegram_user_id`. Бот игнорирует все updates от не-whitelist пользователей (включая ответ боту через тот же токен от другого аккаунта).
- Файлы `media/inbox/` могут содержать фотографии оператора — каталог должен быть в `.gitignore` и иметь права 700 на Mac.
- Логи `logs/inbox-bot-*.log` могут содержать текст заметок — также 700, не коммитятся.

### Координация с другими Telethon-процессами

ADR-010 Задание 3 (G9) планирует `ingestion/tg_incremental.py` — отдельный Telethon long-polling. Бот ADR-016 (G18.2) — **другой** Telethon-клиент, использующий бот-токен (а не user-сессию). Это разные сессии Telegram и они не конфликтуют по API. Но на уровне процессов:

- Каждый Telethon-клиент должен иметь **свою** session-файл (`.session`), не общий.
- Каждый — свой launchd plist (G8 FastAPI / G9 incremental / G18.2 inbox-bot).
- В runbook G18.2 явно указать: «при запуске убедиться, что session-файл бота лежит в отдельном каталоге, не пересекается с G9».

### Что придётся учитывать дальше

- **Измерительный шаг (важно).** До запуска G18 предлагается лёгкий приём: одну неделю оператор в конце дня помечает «сегодня я делал через Cowork: X / Y / Z и это можно было бы дашбордом». Если по итогам недели pending_orders / customer_lookup / pricing — топ-3 — реализуем эти артефакты. Если другое — приоритет артефактов меняется. Без этого можем построить дашборд для несуществующей боли.
- **G14 (полный Telegram-бот) переопределяется.** Бот из ADR-016 — capture-only inbox. Если в будущем оператор захочет полный CRM в Telegram, это G14 — отдельный ADR с полным составом команд и write в основные таблицы. ADR-016 не закрывает G14.
- **Артефакты — отдельная папка.** `crm-mcp/artifacts/` (гит-версионируемая) или встроенные в Cowork через `create_artifact` (живут в Cowork). Решается на pre-flight реализации — у обоих подходов свои плюсы.
- **При активации B1 — обновить `cowork-system-prompt.md`.** Добавить два новых tool-а (`list_inbox_captures`, `mark_inbox_capture`), описать утренний workflow разбора inbox.
- **`inbox_capture` имеет `processed_by_action: created_order:Z-1580`** — это слабая связь, не FK. Достаточно для аудита, но не для join-аналитики. Если возникнет потребность в строгой связи — addendum к ADR-016 с FK на `orders_order` / `orders_customer`.

### Подзадача G18.2 — Mobile capture-only Telegram-бот

Запускается через **Cowork-pf → Claude Code**, Sonnet 4.6 + Medium.

1. **Миграция Alembic** — таблица `inbox_capture`:

   ```sql
   CREATE TABLE inbox_capture (
       id BIGSERIAL PRIMARY KEY,
       captured_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
       source VARCHAR(20) NOT NULL,
       kind VARCHAR(20) NOT NULL,
       text TEXT NULL,
       media_path TEXT NULL,
       status VARCHAR(20) NOT NULL DEFAULT 'pending',
       processed_at TIMESTAMPTZ NULL,
       processed_by_action VARCHAR(40) NULL,
       raw_metadata JSONB NULL,
       CONSTRAINT chk_inbox_kind CHECK (kind IN ('note','lead','photo')),
       CONSTRAINT chk_inbox_status CHECK (status IN ('pending','processed','archived'))
   );
   CREATE INDEX ix_inbox_capture_status_captured ON inbox_capture(status, captured_at DESC);
   ```

2. **SQLAlchemy-модель** — `app/inbox/models.py` (новый модуль). Public `__init__.py` экспорт.

3. **MCP-tools** — `crm-mcp/tools/list_inbox_captures.py` и `crm-mcp/tools/mark_inbox_capture.py` (паттерны из существующих `get_unreviewed_chats` / `link_chat_to_customer`).

4. **Telegram-бот** — `ingestion/inbox_bot.py` (Telethon long-polling, аналог `tg_import.py` структурно). Whitelist `INBOX_BOT_ALLOWED_USER_IDS` через pydantic-settings.

5. **launchd plist** — `~/Library/LaunchAgents/com.pilistrogai.inbox-bot.plist` (паттерн из G8 FastAPI autostart). ASCII-имя, без кириллицы (см. open question от 2026-04-22 про ADR-010).

6. **Runbook** — `docs/runbook_inbox_bot.md`: регистрация бота через @BotFather, как узнать свой `telegram_user_id`, установка `.env` переменных, launchctl команды, где смотреть логи, что делать при сбое.

7. **Тесты:**
   - Юнит-тесты на `list_inbox_captures` и `mark_inbox_capture` (фильтры, статусы, edge cases).
   - Тест на бот: mock Telethon update от `/note текст` → запись появляется в БД.
   - Тест на whitelist: update от чужого `telegram_user_id` → no-op.
