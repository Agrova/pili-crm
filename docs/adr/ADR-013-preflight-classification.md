# ADR-013: Быстрая preflight-классификация чатов перед полным анализом

**Статус:** draft
**Дата:** 2026-04-24
**Связанные ADR:** ADR-011 (Telegram chat analysis pipeline), ADR-012 (Multi-account Telegram support)
**Связанные документы:** `06_open_questions.md`, исторический артефакт `/Users/protey/Downloads/tool-shop-crm/tg_scan_results.json`

---

## Контекст

ADR-011 определил пайплайн анализа Telegram-переписки: четыре фазы (чанкинг → иерархическая саммаризация → narrative → extract JSON) прогоняются для каждого чата. На 303 чата среднего мака без GPU полный прогон оценивается в 10+ часов.

Однако не все чаты — клиентские. По результатам инвентаризации `tool-shop-crm/tg_scan_results.json`:

- **175 чатов** классифицированы как `client` — нужен полный анализ
- **112 `unknown`** — возможные клиенты, нужен анализ
- **8 `friend`**, **4 `service`**, **1 `family`** — заведомо не клиенты
- **90+ пустых чатов** (0 сообщений) — не подлежат анализу вовсе

Итого **минимум 100+ чатов (~33%)** проходят полный анализ впустую. Полный анализ маленького чата на 50 сообщений занимает 3-5 минут на Qwen без GPU. Полный анализ «Жены Анастасии» на 542 сообщения не даст никакого полезного структурированного результата — она не клиент. На выходе пустой narrative и пустой JSON, но Qwen потратил 15+ минут.

**Проблема:** основной пайплайн ADR-011 применяется одинаково ко всем чатам. Нет дешёвого способа отфильтровать заведомо не-клиентские чаты до запуска дорогих фаз 1-4.

**Второе обстоятельство:** в `/Users/protey/Downloads/tool-shop-crm/tg_scan_results.json` уже лежит готовая классификация 303 чатов от прошлого прогона Qwen. Инвентаризация показала хорошее качество классификации. Этот результат можно использовать как стартовую точку, не прогоняя классификатор заново для существующих чатов.

**Решение нужно принять до реализации ADR-011 Задачи 3** (скрипт анализатора). Если сначала написать скрипт без preflight, потом переделывать — это лишние итерации Claude Code.

## Варианты

### Вариант A — Не делать preflight, прогонять всё подряд

Оставить пайплайн ADR-011 как есть. Надеяться на линейное масштабирование времени.

- **Плюсы:** ничего менять не надо; архитектурно проще.
- **Минусы:** ~3-5 часов впустую на первом прогоне; на инкрементальных прогонах эффект накапливается.

### Вариант B — Preflight-классификация одним коротким вызовом Qwen перед фазами 1-4 ✅ выбран

Нулевая фаза: короткий промт к Qwen с первыми/последними сообщениями чата + имя чата. Ответ — классификация (`client` / `possible_client` / `not_client`) с confidence. При `not_client` с `high` confidence — пропускаем полный анализ, записываем минимальную запись в `analysis_chat_analysis`. Во всех остальных случаях — полный пайплайн.

- **Плюсы:** экономит 30-50% времени первого прогона; консервативная политика (при сомнении — полный анализ, не теряем потенциальных клиентов); использует ту же локальную LLM, не вводит новых зависимостей; можно стартовать из `tg_scan_results.json` как из кэша.
- **Минусы:** добавляет фазу в пайплайн; отдельная логика сохранения результата classification; возможна ошибка классификатора (решается консервативностью — при low confidence всегда полный анализ).

### Вариант C — Статистические эвристики без LLM

Правила типа «если 0 сообщений → not_client», «если имя содержит слово из стоп-листа (бот, сервис, жена, мама) → not_client», «если только исходящие сообщения → not_client».

- **Плюсы:** бесплатно, моментально, детерминированно.
- **Минусы:** ограниченный охват (покрывает только очевидные случаи); списки эвристик быстро разрастаются и становятся неуправляемыми; не учитывает контекст переписки.

### Вариант D — Комбинация B + C: сначала эвристики, потом LLM для спорных случаев

Быстрые правила отсекают очевидные случаи (пустые чаты, стоп-лист имён), LLM добирает остальные.

- **Плюсы:** лучшее из обоих подходов: детерминированная экономия на очевидном + LLM для сложных случаев.
- **Минусы:** две системы вместо одной; сложнее поддерживать; часть кода в Python (эвристики), часть в промте.

## Критерии выбора

- **Экономия времени:** должна быть ощутимой на 300+ чатах
- **Надёжность:** не пропустить ни одного реального клиента (false negative хуже false positive)
- **Простота:** минимум новой инфраструктуры; желательно — продолжение того же паттерна что в ADR-011
- **Совместимость с ADR-011:** не ломать существующие фазы и схему `analysis_chat_analysis`
- **Возможность использовать существующий артефакт:** `tg_scan_results.json` содержит уже готовые классификации

## Принятое решение

**Вариант B — LLM preflight одним вызовом Qwen, с возможностью стартовать из `tg_scan_results.json`.**

Вариант D был рассмотрен и отклонён: эвристики на 0 сообщений уже покрыты естественным образом (0 сообщений = nothing to classify, пропускаем без вызова LLM). Расширенный стоп-лист имён («жена», «сервис», «бот») в коде — это та самая «жёсткая логика», от которой оператор явно отказался в обсуждениях ADR-011. Пусть модель решает.

### 1. Фаза 0 пайплайна

Добавляется в `analysis/run.py` перед фазой 1 (чанкинг). Логика:

```
Для каждого чата в работе:
    Если в чате 0 сообщений:
        classification = "empty"
        Записать minimal analysis_chat_analysis и перейти к следующему чату

    Если у чата уже есть valid preflight_classification (см. кэш ниже):
        Использовать его

    Иначе:
        vytbor = выбрать preview_sample(chat)    # см. раздел 2
        result = qwen_classify(vytbor)            # см. раздел 3
        Сохранить result как preflight_classification

    Если result.classification == "not_client" и result.confidence == "high":
        Записать minimal analysis_chat_analysis с пометкой skipped_reason='not_client' и перейти к следующему чату

    Иначе:
        Перейти к фазе 1 (полный анализ)
```

Консервативность: любое `possible_client`, `client`, или `not_client` с `low/medium` confidence → полный анализ. Экономия только на `not_client` + `high`.

### 2. Preview sample — что подаём Qwen для классификации

Состав контекста:

- **Имя чата** (`communications_telegram_chat.title`) — часто уже содержит сигнал («Жена», «Сергей Викторович», «Telegram», «бот такой-то»)
- **Метаданные:** общее число сообщений, число исходящих vs входящих, даты первого и последнего сообщения
- **Первые 5 сообщений** (самые старые) — часто содержат факт знакомства: «здравствуйте, хотел уточнить про ваш зензубель» vs «Привет зай, как дела?»
- **Последние 5 сообщений** (самые свежие) — показывают, как переписка живёт сейчас

Итого на входе ~10-15 сообщений. Для типичного чата это 500-1500 токенов input + ~200 output = 1-3 секунды на Qwen без GPU.

На 303 чата фаза 0 займёт 5-15 минут — приемлемый overhead ради экономии часов полного анализа.

### 3. Промт Qwen и схема ответа

Промт (по-русски, Qwen3-14B работает с русским прилично):

```
Ты помощник оператора магазина столярных инструментов ПилиСтрогай. На входе — превью Telegram-чата из его личной переписки. Твоя задача: определить, клиент ли это магазина.

Имя чата: {title}
Всего сообщений: {total_messages}
Исходящих от оператора: {outgoing_count}
Входящих: {incoming_count}
Первое сообщение: {first_message_date}
Последнее сообщение: {last_message_date}

Первые сообщения:
{first_5_messages}

Последние сообщения:
{last_5_messages}

Верни JSON строго в формате:
{
  "classification": "client" | "possible_client" | "not_client" | "family" | "friend" | "service",
  "confidence": "low" | "medium" | "high",
  "reason": "короткое обоснование"
}

Признаки клиента: спрашивают цены, обсуждают товары, заказы, доставку, инструменты.
Признаки family/friend: личные темы, бытовое общение, без торговли.
Признаки service: автоматические уведомления Telegram, боты, сервисы доставки.
При сомнении — выбирай possible_client с medium confidence.
```

Pydantic-схема ответа:

```python
class PreflightClassification(BaseModel):
    model_config = ConfigDict(extra='forbid')

    classification: Literal["client", "possible_client", "not_client", "family", "friend", "service", "empty"]
    confidence: Literal["low", "medium", "high"]
    reason: str
```

Сохраняется в новых колонках `analysis_chat_analysis` (см. раздел 5).

### 4. Импорт исторических классификаций из `tg_scan_results.json`

Одноразовая импорт-команда — отдельная задача для Prompt Factory.

Скрипт `analysis/import_preflight_from_toolshop.py`:

1. Читает `/Users/protey/Downloads/tool-shop-crm/tg_scan_results.json`
2. Для каждой записи сопоставляет `chat.title` или `chat.telegram_chat_id` с `communications_telegram_chat`
3. Если находит — создаёт запись `analysis_chat_analysis` с заполненными полями preflight (classification, confidence, reason) и пустыми полями полного анализа (`narrative_markdown=''`, `structured_extract={'_v': 1}`)
4. Версия анализатора для этих записей: `v0.9+qwen3-14b-toolshop-legacy` — особая версия, отличная от `v1.0+qwen3-14b` основного пайплайна. Это помечает записи как "preflight уже сделан, полный анализ ещё нет"

После импорта: при прогоне `analysis/run.py` на чате, который уже имеет такую запись, фаза 0 пропускается (используется кэшированная классификация), но фазы 1-4 выполняются как обычно (если классификация не `not_client/high`).

| `unknown` | `possible_client` (конвертируется в более обтекаемое) |
| `friend` | `friend` |
| `family` | `family` |

Confidence — из исходного JSON (там тоже есть такое поле).

### 5. Изменения в схеме `analysis_chat_analysis`

Миграция добавляет новые колонки:

| Поле | Тип | Назначение |
|---|---|---|
| `preflight_classification` | `TEXT NULL` | `client` / `possible_client` / `not_client` / `family` / `friend` / `service` / `empty` |
| `preflight_confidence` | `TEXT NULL` | `low` / `medium` / `high` |
| `preflight_reason` | `TEXT NULL` | Короткое обоснование (диагностика) |
| `skipped_reason` | `TEXT NULL` | Если полный анализ был пропущен — причина (`not_client`, `empty`). Иначе NULL |

Нужны ли NOT NULL constraints на preflight_*? **Нет** — это опциональная фаза. В future-proof смысле могут быть сценарии запуска без preflight (например, при тестах).

Добавить CHECK constraint:

```sql
CHECK (
  -- если skipped_reason заполнен, полный анализ должен быть пустым
  (skipped_reason IS NULL)
  OR (narrative_markdown = '' AND structured_extract = '{"_v": 1}'::jsonb)
)
```

Это защита: нельзя пометить запись как `skipped` и одновременно иметь полный narrative.

### 6. Interaction с Задачей 2 ADR-011 (сервис)

Сервисный метод `apply_analysis_result` (см. задание ADR-011 Задача 2) должен учесть поле `skipped_reason`:

- Если `skipped_reason IS NOT NULL` — **не применять** результат к профилю клиента и не создавать draft-заказы. Записываем только `analysis_chat_analysis`, `orders_customer_profile` не трогаем.

Это маленькая правка в логике сервиса. ADR-011 Задача 2 её учитывает (ниже будет пометка в её задание).

### 7. Возможность прогнать только preflight

Полезно для оператора — быстро увидеть классификацию всех 303 чатов (минут 10-15 на Qwen), принять решения, разобрать очередь модерации с информацией о классификации, и только потом запускать полный анализ клиентских.

CLI-флаги для `analysis/run.py`:

| Флаг | Назначение |
|---|---|
| `--preflight-only` | Выполнить только фазу 0, без фаз 1-4. Полезно для первого прогона. |
| `--skip-preflight` | Пропустить фазу 0, сразу в полный анализ. Полезно если preflight уже сделан отдельно. |
| `--force-preflight` | Перезапустить фазу 0 даже если результат уже есть. Полезно при смене промта preflight. |

По умолчанию: обычный запуск выполняет фазу 0 и фазы 1-4 последовательно, с использованием кэша на обоих уровнях.

### 8. MCP-tool `get_unreviewed_chats` — показ классификации

После выполнения preflight, `get_unreviewed_chats` обогащается новым полем:

## Последствия

### Что становится проще

- **Экономия часов:** первый прогон анализатора на 303 чатах сокращается на 30-50%
- **Быстрая обзорная инвентаризация:** `--preflight-only` позволяет за 15 минут получить картину «кто клиент, кто нет» по всему корпусу
- **Информативная модерация:** оператор видит классификацию при разборе очереди — меньше ручного чтения превью
- **Переиспользование старых результатов:** `tg_scan_results.json` не пропадает, импортируется как кэш preflight'а

### Какие ограничения появляются

- **Отдельная фаза в пайплайне:** +1 уровень сложности в `analysis/run.py`. Требует аккуратной интеграции с чекпоинтами ADR-011 (stage в `analysis_chat_analysis_state` расширяется значением `preflight`)
- **Риск false negative:** preflight может ошибочно пометить клиента как `not_client` с `high` confidence. Смягчается консервативностью (только high блокирует полный анализ; low и medium — анализ продолжается) и возможностью принудительно перепрогнать через `--force-preflight`
- **Зависимость от `tg_scan_results.json`:** если файл недоступен, потерян или оператор работает на чистой установке — нет кэша. Решается тем, что импорт исторического файла — опциональный шаг, не блокер
- **Инкрементальные прогоны:** при первом появлении нового чата (через ADR-010 Задание 3, Telethon) preflight будет его анализировать заново — это ок, это именно его назначение

### Что придётся учитывать дальше

- ADR-011 Задача 3 (скрипт анализатора) — реализует фазу 0 вместе с остальными
- ADR-011 Задача 5 (расширение `get_unreviewed_chats`) — добавляет три поля preflight
- При смене модели (например, Qwen3-14B → Qwen3-32B или другая) — preflight-классификации могут поменяться; нужна политика invalidation (в рамках policy ADR-011 про `analyzer_version`)
