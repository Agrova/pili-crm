# ADR-005: Дублирование данных в читаемые файлы (Google Sheets-зеркало БД)

**Статус:** принят
**Дата:** 2026-04-22
**Связанные ADR:** ADR-001 v2 (модульный монолит), ADR-002 (стек), ADR-003 final-ready + Addendum (схема), ADR-004 (pricing policy)
**Связанные документы:** `docs/cowork-system-prompt.md` (раздел 2 — источник истины)

---

## Контекст

Проект ПилиСтрогай строится вокруг PostgreSQL как единственного источника истины, доступ — через MCP-tools из Cowork (ADR-003, системный промт Cowork, раздел 2). Такая схема надёжна, но имеет ограничения для повседневной работы оператора:

1. **Чтение данных ограничено сессией в Cowork.** Оператор не может бегло просмотреть список заказов вне Cowork — нужен запуск приложения, подключение MCP, формулировка запроса. Для быстрого «взглянуть» это избыточно.
2. **Нет мобильного доступа.** Cowork — desktop-only. Telegram-бот в roadmap, но до его реализации нужен путь «посмотреть с телефона».
3. **Нет внешних выгрузок.** Если оператор хочет показать данные третьему лицу (бухгалтеру, партнёру), поделиться отчётом — нет простого способа, кроме ручного экспорта через SQL.
4. **Нет быстрого поиска в стороннем инструменте.** Excel-фильтры, Ctrl+F, сводные таблицы — привычные оператору инструменты, которых нет в Cowork.

Требуется решить: как дать оператору человекочитаемое представление данных БД, сохранив PostgreSQL единственным источником истины. Решение нужно принять **до** расширения функциональности MCP — иначе возникнет соблазн решать эти задачи через новые MCP-tools, перегружая их нецелевой функциональностью.

## Варианты

### Вариант 1. Markdown-файлы в Obsidian vault

Экспорт БД в набор связанных `.md`-файлов, просматриваемых через Obsidian или любой markdown-редактор.

- **Плюсы:** бесплатно, локально, работает offline, поддерживает связи `[[link]]`, хорошо индексируется.
- **Минусы:** нет таблиц в привычном виде (markdown-таблицы ограничены), нет формул, нет фильтров, нет мобильного доступа без Obsidian Sync (платно). Cowork не получает выигрыша — он уже может читать файлы через MCP, Obsidian лишний слой.

### Вариант 2. Excel (.xlsx) локально

Одна книга с вкладками по таблицам, пересобирается экспортом из БД.

- **Плюсы:** знакомый инструмент, полноценные формулы и фильтры, работает offline, простой код (`openpyxl`).
- **Минусы:** нет мобильного доступа (или неудобно), нет совместного чтения, риск одновременного открытия файла (блокировка), нет встроенной истории изменений.

### Вариант 3. Google Sheets в облаке

Одна книга в Google Drive, пересобирается MCP-сервером через Google Sheets API.

- **Плюсы:** мобильный доступ нативно, совместное чтение из коробки, встроенная история изменений Google, защита листов от правки, не блокируется при открытии.
- **Минусы:** требует интернета, нужна настройка API-авторизации (service account), зависимость от Google.

### Вариант 4. Ничего не делать, расширять MCP-tools

Отказаться от зеркала, вместо этого добавлять в MCP новые tools для специфических выгрузок (отчёты, экспорты в PDF, сводки).

- **Плюсы:** единый канал работы с данными.
- **Минусы:** тормозит повседневную работу (всё через Cowork), не решает мобильный доступ, перегружает MCP нецелевыми функциями, не даёт оператору привычных инструментов анализа.

---

## Критерии выбора

- Сохранение PostgreSQL как единственного источника истины.
- Мобильный доступ к данным без запуска Cowork.
- Привычные инструменты поиска и фильтрации.
- Простота настройки и поддержки.
- Безопасность (защита от случайной правки).
- Возможность совместного чтения третьими лицами.
- Совместимость с текущей архитектурой (Python 3.12+, локальный setup, macOS).

---

## Принятое решение

Принят **Вариант 3 — Google Sheets в облаке**.

### Ключевые параметры

1. **Направление:** файлы — read-only зеркало БД. Источник истины — PostgreSQL. При расхождении файлы перегенерируются из БД.

2. **Один документ, вкладки по таблицам.** Одна Google Sheets-книга, ссылка на которую фиксируется в конфиге MCP-сервера. По одной вкладке на каждую экспортируемую таблицу.

3. **Что экспортируется:**
   - **Всегда (операционные таблицы):** customers, customer_contacts, orders, order_items, catalog_products, catalog_suppliers, stock_items, stock_movements, shipments, shipment_items, и все операционные таблицы communications-модуля по мере их появления.
   - **По необходимости (справочники):** enum-справочники, pricing_constants, currency_rates.
   - **Не экспортируется:** служебные таблицы (alembic_version), audit-таблицы (если появятся), внутренние join-only таблицы.
   - Точный список — в конфиге MCP-сервера, не в ADR. При изменении схемы список обновляется в коде, ADR не переписывается.

4. **Формат вкладок:**
   - **Операционные таблицы — обогащённые.** Вкладка `orders` содержит не `customer_id`, а `Клиент` (ФИО) и `Telegram` (username или ссылка). Статусы — русскими метками (маппинг из системного промта Cowork, раздел 5). Суммы — в рублях с разделителями.
   - **Справочники — нормализованные.** Точная копия таблицы БД, без преобразований.

5. **Триггер обновления (гибридный):**
   - **Автоматический при старте FastAPI** через `lifespan` event. Проверка: `(now() - last_export_at) > 24h AND max(updated_at по операционным таблицам) > last_export_at`. Если обе условия выполнены — экспорт.
   - **Ручной через MCP-tool `refresh_mirror(force=False)`.** По умолчанию проверяет изменения. `force=True` — экспорт безусловный (для случаев ошибок форматирования).
   - **По расписанию (cron / APScheduler) — не используется** для MVP. Старт FastAPI покрывает ежедневный кейс.

6. **Определение «были ли изменения»:** сравнение `max(updated_at)` по операционным таблицам с `last_export_at` из служебной таблицы `mirror_metadata`. Способ самоподдерживающийся — новая операционная таблица автоматически попадает в проверку, если у неё есть поле `updated_at` (стандарт из ADR-003).

7. **Служебная таблица `mirror_metadata`:**
   ```sql
   CREATE TABLE mirror_metadata (
     id INT PRIMARY KEY DEFAULT 1,
     last_export_at TIMESTAMPTZ,
     last_export_status TEXT,            -- 'success' | 'error'
     last_export_error TEXT,              -- NULL если success
     last_export_duration_ms INT,
     row_counts JSONB,                    -- {"customers": 36, "orders": 62, ...}
     CONSTRAINT mirror_metadata_singleton CHECK (id = 1)
   );
   ```
   Одна строка с PK=1. Upsert при каждом экспорте.

8. **Логирование:**
   - Файл `crm-mcp/mirror.log` — подробный лог экспорта (start, duration, errors, API-ответы Google).
   - Таблица `mirror_metadata` — статус последнего экспорта.
   - Уведомлений оператору о провале **нет**. Оператор при сомнении использует MCP-tool `get_mirror_status()` или смотрит лог.

9. **Границы использования (зафиксировано в системном промте Cowork и в README зеркала):**
   - **Можно:** читать, фильтровать, копировать, создавать **собственные вкладки** с пометками (они не перетираются).
   - **Нельзя:** править данные на основных вкладках (при следующем экспорте изменения потеряются), считать зеркало свежее БД при расхождении, использовать зеркало как источник данных для Cowork.
   - **На уровне реализации:** основные вкладки защищены через Protected ranges (Google Sheets API), пользовательские вкладки — без защиты.

10. **Новые MCP-tools:**
    - `refresh_mirror(force: bool = False)` — ручной триггер.
    - `get_mirror_status()` — возвращает содержимое `mirror_metadata` + проверка «есть ли изменения с последнего экспорта».

## Последствия

### Что становится проще

- Оператор может просмотреть заказы, клиентов, остатки без запуска Cowork.
- Мобильный доступ работает сразу через Google Sheets app.
- Делиться данными с третьими лицами — через Share в Google Sheets.
- Привычные фильтры, Ctrl+F, сводные таблицы — доступны нативно.
- Cowork не нагружен функциями отчётности — он про транзакционные операции.

### Какие ограничения появляются

- Появляется зависимость от интернета для обновления зеркала (но не для работы Cowork+БД — они локальные).
- Появляется зависимость от Google API и service account (настройка, ключи, квоты).
- Появляется служебная таблица `mirror_metadata` — одна новая миграция.
- Появляется двухканальное представление данных — нужна дисциплина: БД первична, файл вторичен.
- При долгом offline-периоде зеркало устаревает, но самой работе это не мешает — Cowork идёт в БД.

### Что придётся учитывать дальше

- При добавлении новой операционной таблицы — включить её в список экспорта в коде MCP.
- При изменении маппинга статусов (ADR-003 Addendum) — обновить логику обогащения в экспортёре.
- При появлении audit-таблиц — решить, экспортировать ли их. По умолчанию — нет.
- В перспективе — переход на дельта-обновление (вариант T4 из обсуждения), если суточная пересборка станет узким местом.
- При появлении Telegram-бота — оценить, нужен ли отдельный канал обновления зеркала после действий через бота.

## Что должен сделать Claude Code

Задачи для реализации (передаются в Prompt Factory for Claude Code отдельными промтами):

1. **Миграция Alembic:** создать таблицу `mirror_metadata` со схемой из принятого решения, пункт 7. Инициализировать строку с id=1 в том же миграционном скрипте.

2. **Модуль экспорта в `crm-mcp/mirror/`:**
   - `mirror/exporter.py` — основная логика: чтение из БД, формирование вкладок, запись в Google Sheets через API.
   - `mirror/enricher.py` — денормализация операционных таблиц (подстановка имён клиентов, русских статусов, Telegram-ссылок).
   - `mirror/config.py` — список экспортируемых таблиц, путь к service account, ID Google Sheets-документа.
   - `mirror/metadata.py` — работа с таблицей `mirror_metadata` (upsert, проверка изменений через max(updated_at)).

3. **Интеграция с FastAPI:** `lifespan` event в `app/main.py`, который при старте вызывает проверку-и-экспорт (не блокирует старт при ошибке — лог и продолжаем).

4. **Два новых MCP-tools** в `crm-mcp/server.py`:
   - `refresh_mirror(force: bool = False)` — ручной триггер.
   - `get_mirror_status()` — чтение `mirror_metadata`.

5. **Защита основных вкладок:** через Google Sheets API при создании/обновлении — помечать все экспортируемые листы Protected range. Пользовательские вкладки (те, которых нет в списке экспорта) не трогать.

6. **Логирование:** `crm-mcp/mirror.log` с ротацией, уровни INFO для обычных событий и ERROR для провалов.

7. **Конфигурация:**
   - `GOOGLE_SHEETS_SPREADSHEET_ID` — в `.env`.
   - `GOOGLE_SERVICE_ACCOUNT_JSON_PATH` — в `.env`, путь к ключам.
   - `.env.example` обновить.

8. **README экспортёра:** `crm-mcp/mirror/README.md` с инструкцией по первичной настройке (создание Google Sheets, service account, шаринг документа).

9. **Тесты:**
   - Unit: enricher (денормализация), metadata (upsert-логика, проверка изменений).
   - Integration: реальный экспорт в тестовый Google Sheets (опциональный, за флагом).
   - Mock-тесты MCP-tools `refresh_mirror` и `get_mirror_status`.

## Что проверить вручную

- [ ] Миграция `mirror_metadata` применяется на чистой БД и на БД с seed MVP.
- [ ] Service account создан, Google Sheets расшарен на service account с правами Editor.
- [ ] Первый экспорт при старте FastAPI создаёт все вкладки и заполняет их.
- [ ] Второй старт FastAPI в течение 24h — экспорт пропускается (проверить лог).
- [ ] Второй старт FastAPI через 24h+ без изменений в БД — экспорт пропускается.
- [ ] После `create_order` через Cowork — следующий старт FastAPI делает экспорт (max(updated_at) сработал).
- [ ] `refresh_mirror(force=False)` без изменений возвращает «изменений нет».
- [ ] `refresh_mirror(force=True)` выполняет экспорт безусловно.
- [ ] `get_mirror_status()` возвращает корректный timestamp, статус, row_counts.
- [ ] Защита листов работает: ручная правка ячейки на вкладке `orders` блокируется.
- [ ] Пользовательская вкладка (добавленная оператором вручную) не перетирается при экспорте.
- [ ] Русские статусы в вкладке `orders` соответствуют маппингу из системного промта Cowork.
- [ ] Telegram-ссылки в вкладке `customers` формируются по правилу раздела 6 системного промта.
- [ ] При ошибке Google API (например, отозван service account) — FastAPI стартует, `mirror_metadata.last_export_status` = 'error', лог содержит причину.
- [ ] Мобильное приложение Google Sheets показывает данные корректно.

## Открытые вопросы (передаются в `06_open_questions.md`)

- **Q (для ADR-005 в будущем):** когда переходить на дельта-обновление (T4), если суточная пересборка станет узким местом. Критерий: время экспорта >30 секунд или >5 API-вызовов Google в минуту.
- **Q (для архитектурного чата):** нужен ли экспорт audit-таблиц, когда они появятся. По умолчанию — нет.
- **Q (для Prompt Factory при реализации):** какая именно библиотека Google Sheets API — `gspread` (проще) или `google-api-python-client` (официальная, гибче). Решается при написании промта.
