# ADR-003 Addendum: Статус заказа `quoted`

**Статус:** принят
**Дата:** 2026-05-05
**Расширяет:** ADR-003 final-ready, ADR-003 Addendum (статусы), ADR-006 (derive-status trigger)
**Связанные:** ADR-018 (калькулятор нового заказа G19)

---

## Контекст

ADR-003 Addendum зафиксировал 8 статусов заказа: `draft → confirmed → in_procurement → ... → delivered + cancelled`. На практике обнаружился промежуточный сценарий, который не покрывается:

- Оператор делает расчёт нового заказа в калькуляторе (G19), формирует текст для клиента и отправляет его в Telegram.
- Клиент **думает несколько часов или дней** перед тем как согласиться или отказаться.
- В это время заказ существует — его состав, цены и текст уже зафиксированы — но он **не подтверждён** клиентом и не **обрабатывается** оператором (не в работе у поставщика, не в логистике).

Сейчас такой «отправленный, ждём ответа» заказ может оказаться:
- В `draft` — но это семантически неправильно: `draft` — это «оператор ещё работает над заказом, состав не финален». Здесь же состав финален, текст отправлен.
- Сразу в `confirmed` — но это лжёт: клиент не подтвердил.

Накопление таких квази-черновиков уже наблюдалось при разборе анализа чатов (ADR-017, проблема с пустыми draft-заказами, см. v15 PLAN.md). Семантическая граница между «черновик в работе у оператора» и «отправлено клиенту, ждём» нужна явно.

Что **не обсуждается** в этом аддендуме: семантика существующих статусов (без изменений), derive-status trigger для активной фазы (без изменений), TTL и автоматическая отмена (открытый вопрос — см. ниже).

---

## Решение

### Новый статус `quoted` в order-level enum

Добавляется одно значение между `draft` и `confirmed`:

```
draft → quoted → confirmed → in_procurement → shipped_by_supplier
→ received_by_forwarder → arrived → delivered
+ cancelled (из любого состояния)
```

| Значение | Описание | Когда устанавливается |
|---|---|---|
| `draft` | Черновик в работе у оператора, состав может меняться | Создание заказа без указания статуса |
| **`quoted`** | **Расчёт отправлен клиенту, ждём ответа** | **Калькулятор G19 при «создать черновик» — единственный путь сейчас** |
| `confirmed` | Заказ подтверждён клиентом | Оператор переводит вручную после согласия клиента |
| `in_procurement` | Заказан у поставщика (derivation) | Автоматически по item-level статусам |
| ... | ... | ... |

Item-level enum **не меняется** — `quoted` существует только на уровне заказа, потому что у позиций ещё нет операционной активности.

### Русская метка

| Enum | Русская метка |
|---|---|
| `quoted` | Предложено клиенту |

Добавляется в `cowork-system-prompt.md` раздел 5 (mapping таблица).

### Взаимодействие с derive-status trigger (ADR-006)

`quoted` — **ручной** статус. Derive-status trigger не активируется для него.

Логика триггера сейчас (упрощённо):
- Если у заказа есть активные позиции с одним из item-level статусов — заказ переводится в соответствующий order-level статус по mapping.
- Если все позиции `cancelled` — заказ `cancelled`.
- Если позиций нет — статус не меняется (остаётся как был).

С появлением `quoted` правило расширяется одной фразой:
- **Если order.status ∈ {`draft`, `quoted`} — derive-trigger НЕ перезаписывает статус**, даже если у заказа уже есть позиции с item-level статусами. Только ручной переход в `confirmed` (через MCP `update_order(status='confirmed')` или специальный tool `confirm_quote`) разблокирует автоматическую derivation.

Эта оговорка нужна, потому что калькулятор G19 создаёт заказ сразу с позициями (item-level=`pending`). Без этой защиты trigger мгновенно после создания переведёт `quoted → in_procurement`, что неверно (клиент ещё не согласился).

### Workflow перехода

```
Калькулятор G19 → quoted
   │
   ├── Клиент согласился → оператор: confirm_quote(order_id) или update_order(status='confirmed')
   │      → confirmed → derive-trigger активен → in_procurement (автоматически по pending items)
   │
   ├── Клиент отказался → оператор: cancel_order(order_id, reason='customer_refusal')
   │      → cancelled
   │
   └── Клиент молчит → ручная отмена оператором по своему усмотрению
          → cancelled
```

### Защита от пустых quoted

Триггер `_is_actionable_order` из ADR-017 распространяется и на `quoted`: пустой заказ (без позиций) или заказ только с `cancelled`/`delivered` позициями автоматически не попадает в результаты apply_analysis. Калькулятор G19 всегда создаёт заказ с непустым составом — пустых `quoted` не будет.

## Что **не** включено в этот аддендум

### TTL и автоматическая отмена `quoted → cancelled`

В первоначальном обсуждении предлагалось: «через 30 дней без ответа клиента — автоматический переход в `cancelled`». Это **отложено** до открытого вопроса:

- Слишком жёстко — клиент может вернуться через 2 месяца.
- Требует отдельного scheduler-job (APScheduler).
- Лучше дождаться реальной статистики «висячих` quoted` после 1-2 месяцев работы калькулятора и принять addendum по факту.

См. open question (создать после принятия этого аддендума).

### Маппинг истории analysis → quoted

ADR-017 ввёл `_is_actionable_order` для фильтрации исторических заказов из анализа. Должен ли apply_analysis создавать `quoted` вместо `confirmed` если в источнике (Telegram-чате) заказ обсуждается, но статус неясен? **Нет, в этом аддендуме не меняем поведение apply_analysis.** apply_analysis продолжает создавать заказы со статусом по логике ADR-017. `quoted` — только для калькулятора G19.

## Что должен сделать Claude Code

Реализация происходит **внутри G19** (см. ADR-018), не отдельной задачей. Минимальный набор:

1. **Миграция Alembic:**
   ```sql
   ALTER TYPE orders_order_status ADD VALUE 'quoted' BEFORE 'confirmed';
   ```
   ALTER TYPE ... ADD VALUE требует non-transactional миграции в PostgreSQL — учесть в Alembic-операции (`op.execute` с `autocommit_block` или отдельная миграция вне транзакции).

2. **Обновление derive-status trigger (ADR-006):**
   - В функции derive-status добавить guard: `IF NEW.status IN ('draft', 'quoted') THEN RETURN NEW; END IF;` — тригер ничего не делает.
   - Тест: создать заказ со статусом `quoted` и одной `pending` позицией — статус должен остаться `quoted`, не перейти в `in_procurement`.

3. **Обновление `cowork-system-prompt.md` раздел 5:** добавить строку `| quoted | Предложено клиенту |` в таблицу маппинга.

4. **Новый MCP-tool `confirm_quote(order_id)`** (или расширение `update_order` принимать `status='confirmed'` явно): переводит `quoted → confirmed`, разблокирует derive-trigger, который автоматически вычислит `in_procurement` по составу.

5. **Тесты:**
   - Создание заказа `quoted` с непустым составом — статус не меняется автоматически.
   - `confirm_quote` переводит в `confirmed` и derive-trigger переводит в `in_procurement`.
   - `cancel_order` для `quoted` работает как обычно.
   - Ошибка попытки перехода `arrived → quoted` (нельзя откатиться назад).
