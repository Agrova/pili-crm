# tool-gaps.md — пробелы в функциональности MCP

**Назначение:** фиксация случаев, когда среди существующих 9 MCP-tools не хватает нужного для типовой операции оператора.

**Куда НЕ писать сюда:**
- Эргономика существующих tools → `crm-mcp/IMPROVEMENTS.md`
- Отсутствие поля / статуса / сущности в БД → `docs/schema-gaps.md`

См. decision tree в `docs/cowork-system-prompt.md`, раздел 8.

**Путь решения:**
- Если сущности для нового tool-а уже есть в БД → промт в Prompt Factory for Claude Code.
- Если нужна новая сущность → сначала ADR (через `schema-gaps.md`), потом Prompt Factory.

**Важно:** «нужен tool» ≠ «операция невозможна». Сначала проверь, нет ли уже комбинации существующих tools, которая решает задачу. Если решается в 1-2 шага — записи не нужно. Если требует 3+ шагов или комбинация неочевидна — записываем.

---

## Формат записи

```markdown
## YYYY-MM-DD — короткий заголовок

- **Severity:** low | medium | high
- **Источник:** типовая операция оператора / сценарий
- **Проблема:** какая операция затруднена / невозможна
- **Сценарий:** конкретный пример из практики (1-2 предложения)
- **Предложение:** какой tool добавить, с какими параметрами и возвращаемым значением
- **Зависимости:** нужны ли новые сущности (→ ссылка на запись в `schema-gaps.md`)
- **Статус:** open
- **Связанные решения:** (заполняется позже — ADR-XXX / промт в Prompt Factory)
```

### Severity

- **high** — операция блокирует ежедневную работу, делается полностью вне системы.
- **medium** — операция требует обходного пути через комбинацию tools.
- **low** — было бы удобно иметь, но обходится легко.

## Пример записи (удалить при первой реальной записи)

## 2026-04-22 — нет tool-а для просмотра истории статусов позиции

- **Severity:** medium
- **Источник:** сценарий «клиент спрашивает, когда товар был отправлен»
- **Проблема:** существующие tools показывают только текущий статус позиции. Нет способа увидеть хронологию переходов (когда был `ordered`, когда `shipped`, когда `at_forwarder`).
- **Сценарий:** клиент спросил «когда мой товар отправили». Оператор знает только текущий статус, дату перехода в `shipped` восстановить не может.
- **Предложение:** tool `get_order_item_history(item_id)` → возвращает список переходов (status, timestamp, optional note).
- **Зависимости:** требует таблицы логирования изменений статусов — её в схеме ADR-003 нет → запись в `schema-gaps.md`.
- **Статус:** open
- **Связанные решения:** — (требуется ADR на логирование, затем tool)

## 2026-04-22 — нет tool-а для создания товара в каталоге

- **Severity:** high
- **Источник:** сценарий «пришёл товар, которого нет в каталоге»
- **Проблема:** `add_to_stock` требует, чтобы товар уже существовал в `catalog_product`. Если товара нет — операция блокируется. Создать товар через MCP невозможно.
- **Сценарий:** оператор хотел добавить Festool TS 55 на склад через `add_to_stock` — MCP вернул ошибку «не найден в каталоге». Обходного пути нет.
- **Предложение:** tool `create_product(name, supplier, declared_weight, sku?)` → создаёт запись в `catalog_product` и возвращает `product_id`. Можно совместить с `add_to_stock` через параметр `create_if_missing: true`.
- **Зависимости:** сущность `catalog_product` уже есть в схеме — новая сущность не нужна.
- **Статус:** triaged
- **Связанные решения:** Prompt Factory, Пакет β

## 2026-04-30 — Нет tool для обновления данных клиента

- **Severity:** medium
- **Источник:** create_customer / update workflow
- **Проблема:** нет MCP-tool для обновления существующего клиента (имя, телефон, Telegram). Есть только `create_customer` (создание) и `find_customer` (поиск).
- **Сценарий:** клиент Воропаев (id=250) нужно обновить: добавить имя "Дмитрий", телефон +79165104700, Telegram @Dmitriy Voropaev. Невозможно сделать через MCP.
- **Предложение:** добавить `update_customer(customer_id, name?, phone?, telegram_id?, email?)` tool.
- **Зависимости:** сущность `orders_customer` уже есть в схеме — новая сущность не нужна.
- **Статус:** done
- **Связанные решения:** G5 — реализован `crm-mcp/tools/update_customer.py`, коммит `c6e5112`

## 2026-04-30 — Нет tool для редактирования существующего заказа

- **Severity:** high
- **Источник:** create_order / edit workflow
- **Проблема:** нет MCP-tool для редактирования заказа после создания. Невозможно добавить позицию доставки, изменить цену товара, пересчитать итого, добавить скидку в позицию.
- **Сценарий:** заказ З-1580 создан на $64.90, но нужно добавить доставку $12.90 и учесть скидку $6.49. Невозможно отредактировать через MCP — приходится пересоздавать заказ.
- **Предложение:** добавить `update_order(order_id, items_to_add?, items_to_remove?, price_adjustments?)` tool или хотя бы возможность добавлять новые позиции в существующий заказ.
- **Зависимости:** сущности `orders_order`, `orders_order_item` уже есть — новая сущность не нужна.
- **Статус:** done (частично — итерация 1: items_to_add; items_to_remove и price_adjustments → IMPROVEMENTS.md, итерация 2)
- **Связанные решения:** G5 — реализован `crm-mcp/tools/update_order.py`, коммит `c6e5112`

## 2026-04-30 — Нет tool для просмотра/обновления расчётного курса валют

- **Severity:** medium
- **Источник:** сценарий «обновление курса USD при изменении рыночных условий», расчёт цены при создании заказа
- **Проблема:** курс архитектурно хранится в БД (таблица `pricing_exchange_rate`), но нет MCP-tools для (а) показа актуального курса оператору, (б) добавления новой записи курса. Оператор сейчас обходится черновиком в файле (`pricing-context.md`, удалён 2026-04-30) — не источник истины. При расчёте цены через формулу из ADR-004 курс должен браться из таблицы; неизвестно, есть ли там актуальная запись на сегодняшнюю дату.
- **Сценарий:** оператор знает, что текущий курс USD = 80 RUB. Хочет (1) увидеть, какой курс сейчас зафиксирован в системе, (2) обновить если устарел. Также при создании заказа в Cowork оператор хочет видеть, по какому курсу будет считаться цена.
- **Предложение:**
  - `get_current_exchange_rate(currency: str = "USD") -> dict` — возвращает последнюю запись `pricing_exchange_rate` для валюты (rate, дата, кто добавил)
  - `set_exchange_rate(currency: str, rate: Decimal, note: str?) -> dict` — добавляет новую запись (записи иммутабельны, история сохраняется)
  - Опционально — обогатить `search_products` или `create_order` отображением курса, который применится
- **Зависимости:** таблица `pricing_exchange_rate` уже есть в схеме (ADR-003) — новая сущность не нужна. Нужно убедиться, что в продакшен-БД есть актуальная запись на 2026-04-30.
- **Статус:** open
- **Связанные решения:** возможно объединить с G6 (finance ledger, там есть `finance_exchange_rate`) или сделать отдельным микропромтом. Курс актуален для всех будущих расчётов цен.

## 2026-04-30 — Нет health-check tool для проверки состояния инфраструктуры (Docker, Postgres, FastAPI)

- **Severity:** high
- **Источник:** оперативный workflow — каждый сеанс работы с Cowork
- **Проблема:** при перезагрузке Mac Docker и Postgres-контейнер не стартуют автоматически (Docker Desktop без autostart). FastAPI/MCP-сервер тоже не запущен. Оператор обнаруживает это только когда первый MCP-tool падает с ошибкой. Нет единой точки проверки «система готова к работе».
- **Наблюдение (2026-04-30):** после перезагрузки Mac — Docker Desktop не запустился → контейнер `pili-crm-postgres-1` не стартовал → все MCP-tools вернули `OSError`. Потребовалось несколько шагов ручной диагностики.
- **Предложение:** tool `system_health()` → проверяет и возвращает статус:
  - Docker Desktop запущен (да/нет)
  - Контейнер `pili-crm-postgres-1` running (да/нет)
  - PostgreSQL отвечает на запросы (да/нет)
  - FastAPI `GET /health` → ok (да/нет)
  - Опционально: команды для исправления каждого сбоя (`docker-compose up -d`, `python3 -m uvicorn ...`)
- **Зависимости:** требует системного доступа (shell) из MCP-сервера — не через БД. Альтернатива: отдельный shell-скрипт `scripts/check_health.sh` + alias `crm-health` в `.zshrc`.
- **Статус:** open
- **Связанные решения:** частично перекрывается с G8 (FastAPI launchd autostart) — если Docker и FastAPI стартуют автоматически, проблема исчезает. До реализации G8 — скрипт как workaround.
## 2026-04-30 — Нет tool для применения уже готового анализа к клиенту после привязки чата

- **Severity:** high
- **Источник:** сценарий «link_chat_to_customer → apply накопленных анализов»
- **Проблема:** `analysis/run.py` с флагом `--no-apply` сохраняет `structured_extract` в БД, но не создаёт identity-записи (не вызывает `apply_analysis_to_customer`). Когда оператор позже привязывает чат к клиенту через `link_chat_to_customer`, уже готовые анализы не применяются автоматически — нет ни MCP-tool, ни CLI-команды для этого.
- **Сценарий:** chat 6485 (@vyashin86) проанализирован на PC с `--no-apply`. Чат не был привязан к клиенту Слава Яшин (id=1688). После ручной привязки через `link_chat_to_customer` identity-записи в карантин не попали — нужно вручную перезапускать анализ с `--force` или запускать кастомный Python-скрипт.
- **Предложение:**
  - MCP-tool `apply_pending_analysis(chat_id)` — ищет последний завершённый `analysis_chat_analysis` для чата, вызывает `apply_analysis_to_customer(force=True)`, возвращает результат (identity_quarantined, orders_created, etc.)
  - Или расширить `link_chat_to_customer` параметром `apply_existing_analysis: bool = True` — при привязке автоматически применять последний готовый анализ если есть
  - Также полезен флаг CLI: `python3 -m analysis.run --apply-only --chat-id-range N..N`
- **Зависимости:** `apply_analysis_to_customer` в `app/analysis/service.py` уже реализован, `force=True` поддерживается — новая логика не нужна, только обёртка.
- **Статус:** done
- **Связанные решения:** G5 — реализован `crm-mcp/tools/apply_pending_analysis.py`, коммит `c6e5112`

## 2026-04-30 — Нет tool для просмотра и верификации draft-заказов из анализа

- **Severity:** high
- **Источник:** сценарий G4 — верификация результатов apply_pending_analysis
- **Проблема:** `apply_pending_analysis` создаёт заказы со статусом `draft` (origin=analysis). `pending_orders` их не показывает — фильтрует только `confirmed` и выше. Оператор не может увидеть, верифицировать или отклонить черновики через Cowork.
- **Сценарий:** chat 6485 (Слава Яшин) — apply_pending_analysis создал 26 draft-заказов. `pending_orders` показал только 2 активных. 26 черновиков недоступны оператору.
- **Предложение:**
  - `list_draft_orders(customer_id?)` — список заказов со статусом `draft` и origin=analysis, с позициями. Фильтр по клиенту опциональный.
  - `verify_draft_order(order_id, action: confirm|reject)` — перевести черновик в `confirmed` или удалить как false-positive.
- **Зависимости:** таблицы `orders_order`, `orders_order_item` уже есть — новая сущность не нужна. Нужно убедиться что поле `origin` или аналог есть в схеме.
- **Статус:** done
- **Связанные решения:** G5.5 — реализованы `crm-mcp/tools/list_draft_orders.py` и `crm-mcp/tools/verify_draft_order.py`, `cowork-system-prompt.md` v2.1, 6 тестов в `tests/crm_mcp/test_draft_orders.py`. Закрыто 2026-04-30, CP8.5.

## 2026-04-30 — Нет tool для просмотра narrative + истории клиента из анализа

- **Severity:** medium
- **Источник:** ADR-017 — фильтр исторических orders убирает их из `orders_order`, но они остаются в `analysis_chat_analysis.structured_extract` как ценный read-only контекст
- **Проблема:** narrative_markdown и `extract.orders` (включая отфильтрованные исторические) не доступны оператору через Cowork. После ADR-017 это становится особенно заметно: оператор знает, что фильтр срезал ~64% orders, но не может посмотреть, что именно было срезано — для оценки качества фильтра и для общего контекста по клиенту.
- **Сценарий:** оператор работает с клиентом Слава Яшин (id=1688). Видит 1-2 актуальных draft из G4.6, хочет посмотреть «а что вообще LLM узнал про этого клиента из переписки» — narrative с разделами «Клиент / История / Предпочтения / Доставка / Заказы». Сейчас доступа нет.
- **Предложение:**
  - `get_customer_history(customer_id, chat_id?)` — возвращает narrative_markdown последнего analysis + полный `structured_extract.orders` (включая отфильтрованные) с пометкой какие применены/отфильтрованы. Если `chat_id` не указан — собирает по всем привязанным чатам клиента.
  - Альтернатива (компактнее): `get_chat_narrative(chat_id)` — только narrative + summary structured_extract одного чата.
- **Зависимости:** таблица `analysis_chat_analysis` уже есть (ADR-011). Связь chat ↔ customer через `communications_link` (ADR-011 addendum-1).
- **Статус:** open
- **Связанные решения:** ADR-017 (увеличивает важность tool), G4.6 (триггер на bump v1.5 — `06_open_questions.md` от 2026-04-30 — для оценки нужны narrative + extract).
