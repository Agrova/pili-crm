## Кто пишет в файл

**Операционный Cowork** (см. `cowork-system-prompt.md` раздел 11). Cowork сам определяет момент записи по эвристикам из промта и пишет append-only через файловый доступ. Оператор вручную ничего не заполняет, но может попросить «запиши в журнал измерений» в любой момент — Cowork добавит запись с пометкой `manual=true`.

## Что считается сигналом

**Записываем (содержательные запросы):**
- Запрос с конкретным предметом: «покажи pending Воропаева», «найди Кристину», «сколько сейчас курс».
- Запрос, который требует от Cowork пересказа большого ответа MCP (>3 строк): полный список pending, история клиента, поиск товара.
- Повтор такого же запроса в одной сессии (сильный сигнал — кандидат на дашборд).
- Запрос, который чаще задавался бы вне Mac (мобильный capture).

**Не записываем (шум):**
- Подтверждения «да/нет», «ок», «отмена» — это часть правила двух подтверждений.
- Команды на write-операцию без контекста: «создай заказ», «переведи в shipped» — даже если они идут через MCP-tool, это не lookup.
- Внутренние Cowork-операции (запись в IMPROVEMENTS.md, перечитывание промта).
- Ответы на уточняющие вопросы Cowork («какой Иван», «о каком заказе»).

В пограничных случаях — лучше записать. Шум при разборе фильтруется, пропуск — нет.

## Формат записи

Две независимые секции — `## Request log` и `## Capture log`. Записи append в конец соответствующей секции.

### Формат записи в Request log

```
- [YYYY-MM-DD HH:MM] category=<lookup|history|reference|status_check>
  intent: <одно предложение что искал оператор>
  tools: <какие MCP-tools вызвал, через запятую>
  response_size: <small (≤3 строк) | medium (4-15 строк) | large (>15 строк)>
  could_be_artifact: <yes | no | maybe>
  recurring: <0 | N>  # сколько раз похожий запрос уже был в этой сессии
  manual: <true | false>  # true если оператор попросил «запиши»
```

Категории:
- `lookup` — поиск факта (клиент, товар, цена, статус).
- `history` — просмотр истории/списка (заказы клиента, последние коммуникации, pending-список).
- `reference` — справочник (курс, маржа, политика, шпаргалка).
- `status_check` — «что в работе сейчас», «что застряло» — обзор оперативного состояния.

`response_size` оценивается на глаз по тому, сколько Cowork пересказал бы в чат.

`could_be_artifact`:
- `yes` — типовой запрос, ответ табличный или фиксированной структуры, повторяется → артефакт точно полезен.
- `maybe` — может быть артефактом, но запрос скорее разовый.
- `no` — запрос требует рассуждений Cowork (соединить факты, сделать вывод), не сводится к таблице.

### Формат записи в Capture log

```
- [YYYY-MM-DD HH:MM]
  context: <короткое описание момента: «вспомнил утром», «пришёл с встречи», «на обеде»>
  what_to_capture: <что хотел/надо было зафиксировать>
  was_lost: <true | false>  # true если момент уже забыт частично или окончательно
  source: <claimed_by_operator | inferred_by_cowork>
```

`source`:
- `claimed_by_operator` — оператор сам сказал «вчера хотел записать X, забыл».
- `inferred_by_cowork` — Cowork увидел в начале сессии типовой паттерн «надо было ещё».
