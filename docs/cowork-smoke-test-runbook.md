# Cowork + MCP — Smoke-test runbook

**Назначение:** пошаговая проверка всех 9 MCP-tools через Cowork перед переходом в ежедневную работу. Runbook выполняется вручную оператором.
**Расположение:** `docs/cowork-smoke-test-runbook.md`
**Версия:** 1.0
**Связанные документы:** `docs/cowork-system-prompt.md`, `crm-mcp/` (реализация tools)

---

## Предусловия

До запуска runbook убедиться:

- [ ] PostgreSQL запущен, миграции применены (ADR-003 + Addendum + ADR-004).
- [ ] Seed MVP загружен (36 клиентов, 62 заказа, 133 позиции).
- [ ] MCP-сервер `crm-mcp` запущен и отдаёт список из 9 tools.
- [ ] Cowork подключён к MCP, видит все 9 tools в интерфейсе.
- [ ] Системный промт `docs/cowork-system-prompt.md` загружен в Cowork.
- [ ] Созданы пустые файлы аудита: `crm-mcp/IMPROVEMENTS.md`, `docs/schema-gaps.md`, `docs/tool-gaps.md`.
- [ ] У Cowork есть доступ на запись к этим трём файлам.

Если какой-то пункт не выполнен — сначала закрой его в чате «Настройка Cowork», потом возвращайся сюда.

---

## Как проходить runbook

- **Один сценарий — один тест.** Не смешивай.
- **После каждого сценария** — отметь результат (✅ / ⚠️ / ❌) в чеклисте ниже и, если есть замечание, сразу зафиксируй в нужный аудит-файл (IMPROVEMENTS.md / schema-gaps.md / tool-gaps.md) по decision tree из системного промта.
- **Критерий прохождения:** Cowork выполняет tool, возвращает ожидаемый результат, представляет его с русскими статусами и Telegram-ссылками (где применимо).
- **Критерий замечания:** tool работает, но что-то неудобно/неясно/медленно.
- **Критерий провала:** tool падает, возвращает не то, или Cowork неправильно его интерпретирует.

## Tool 5 — `match_shipment`

**Назначение:** сопоставление поступления с заказами/позициями.

> ⚠️ **Перед тестом:** уточнить через код `crm-mcp/`, является ли tool read-only или пишет связь в БД. От этого зависит требование подтверждения. См. системный промт, раздел 7.

### Сценарий 5.1 — Happy path

**Запрос:** «Пришёл shipment <реальный track или описание из seed>, с чем сопоставить?»

**Ожидается:**
- Список кандидатов (заказы/позиции) с оценкой уверенности.
- Только позиции в статусах `pending`, `ordered`, `shipped`, `at_forwarder`.
- Cowork не делает выбор за оператора — показывает кандидатов, ждёт решения.
- Если tool пишет в БД — правило двух подтверждений сработало.

**Результат:** ☐ ✅ / ☐ ⚠️ / ☐ ❌
**Замечания:**

### Сценарий 5.2 — Edge: нет кандидатов

**Запрос:** сопоставление с данными, которые заведомо ни к чему не подходят.

**Ожидается:** Cowork говорит «нет подходящих позиций», не предлагает нерелевантное.

**Результат:** ☐ ✅ / ☐ ⚠️ / ☐ ❌

## Tool 6 — `create_customer`

**Назначение:** создание нового клиента.

> ⚠️ Write-операция, правило двух подтверждений обязательно. Обязательный pre-check через `find_customer`.

### Сценарий 6.1 — Happy path с проверкой дубля

**Запрос:** «Создай клиента Тестовый Тест, +7 900 000 0001, Telegram @test_cowork_1».

**Ожидается:**
- Cowork сначала вызывает `find_customer` для проверки.
- Показывает сводку: имя, телефон, Telegram.
- Ждёт подтверждение «да» / «ок».
- Только после подтверждения вызывает `create_customer`.
- Сообщает ID нового клиента.

**Результат:** ☐ ✅ / ☐ ⚠️ / ☐ ❌
**Замечания:**

### Сценарий 6.2 — Edge: попытка создать дубль

**Запрос:** «Создай того же самого клиента ещё раз».

**Ожидается:** Cowork обнаруживает дубль через `find_customer` и предлагает работать с существующим, а не создавать второго.

**Результат:** ☐ ✅ / ☐ ⚠️ / ☐ ❌
**Замечания:**

### Сценарий 6.3 — Edge: отказ в подтверждении

**Запрос:** «Создай клиента Отказник, +7 900 000 0002» → на подтверждение ответить «отмена».

**Ожидается:** Cowork не вызывает `create_customer`. Спрашивает что изменить.

**Результат:** ☐ ✅ / ☐ ⚠️ / ☐ ❌

## Tool 9 — `update_order_item_status`

**Назначение:** изменение статуса позиции заказа.

> ⚠️ Подтверждение только при переходе в `cancelled` или `delivered`. Остальные переходы — сразу.

### Сценарий 9.1 — Happy path, штатный переход

**Запрос:** «Переведи позицию <ID> в статус "заказан у поставщика"».

**Ожидается:**
- Cowork конвертирует русскую метку в enum `ordered`.
- Выполняет tool сразу (подтверждение не требуется).
- Сообщает новый статус позиции и (если изменился) новый статус заказа по derivation rule.

**Результат:** ☐ ✅ / ☐ ⚠️ / ☐ ❌
**Замечания:**

### Сценарий 9.2 — Подтверждение при delivered

**Запрос:** «Позиция <ID> передана клиенту».

**Ожидается:**
- Cowork просит подтверждение (переход в `delivered`).
- После подтверждения — статус обновлён.
- Если это последняя активная позиция заказа — статус заказа тоже `delivered`.

**Результат:** ☐ ✅ / ☐ ⚠️ / ☐ ❌
**Замечания:**

### Сценарий 9.3 — Подтверждение при cancelled

**Запрос:** «Отмени позицию <ID>».

**Ожидается:**
- Cowork просит подтверждение (переход в `cancelled`).
- После подтверждения — статус обновлён.
- Derivation rule корректно пересчитывает статус заказа (активные позиции, если есть).

**Результат:** ☐ ✅ / ☐ ⚠️ / ☐ ❌
**Замечания:**

### Сценарий 9.4 — Derivation rule: все позиции cancelled

**Запрос:** отменить все позиции заказа по очереди.

**Ожидается:** когда отменена последняя — статус заказа = `cancelled`.

**Результат:** ☐ ✅ / ☐ ⚠️ / ☐ ❌

## Итоговый чек после runbook

- [ ] Все 9 tools вызваны хотя бы в одном сценарии.
- [ ] Все обнаруженные замечания записаны в один из трёх аудит-файлов.
- [ ] Правило двух подтверждений ни разу не нарушено.
- [ ] Cowork ни разу не придумал данных.
- [ ] Derivation rule работает корректно.
- [ ] Русские статусы используются везде в UI, английские enum — везде в MCP-вызовах.
- [ ] Telegram-ссылки формируются по правилу.

Если все галочки — Cowork+MCP готов к ежедневной работе.

Если есть провалы (❌) — фиксируй в `IMPROVEMENTS.md` или `tool-gaps.md` / `schema-gaps.md` в зависимости от природы проблемы.
