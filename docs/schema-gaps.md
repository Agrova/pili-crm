**Назначение:** фиксация случаев, когда в существующей модели данных не хватает поля, статуса, enum-значения, связи или целой сущности для отражения бизнес-реальности.

- Эргономика существующих MCP-tools → `crm-mcp/IMPROVEMENTS.md`
- Отсутствие tool-а для уже существующей сущности → `docs/tool-gaps.md`

См. decision tree в `docs/cowork-system-prompt.md`, раздел 8.

## 2026-04-30 — Отсутствует таблица `pricing_price_calculation`

- **Severity:** high
- **Источник:** `apply_pending_analysis` / чат 6485 (@vyashin86)
- **Проблема:** `orders_order_item.price_calculation_id` ссылается на таблицу `pricing_price_calculation`, которая не существует в БД — миграция не была применена или таблица не создавалась.
- **Сценарий:** при попытке apply_pending_analysis для клиента Слава Яшин (id=1688) получена ошибка `NoReferencedTableError` с указанием на отсутствующую таблицу.
- **Предложение:** проверить наличие миграции для `pricing_price_calculation` и применить её через Alembic. Если миграция не создана — разработать и применить.
- **Статус:** open
- **Связанные решения:** —

**Путь решения:** обсуждение в архитектурном чате → ADR → миграция Alembic → возможно правка MCP-tools.

**Важно:** запись здесь — это сигнал, а не решение. Решение принимается в архитектурном чате как ADR. До принятия ADR запись остаётся в статусе `open`.

## Формат записи

```markdown
## YYYY-MM-DD — короткий заголовок

- **Severity:** low | medium | high
- **Источник:** таблица / сущность / сценарий оператора
- **Проблема:** какой факт нельзя зафиксировать / чего не хватает
- **Сценарий:** конкретный пример из практики (1-2 предложения)
- **Предложение:** какое поле / статус / сущность добавить
- **Статус:** open
- **Связанные решения:** (заполняется позже — ADR-XXX)
```

### Severity

- **high** — важный бизнес-факт нельзя зафиксировать, теряются данные.
- **medium** — факт фиксируется обходным путём (например, в notes-поле), но теряется структурированность.
- **low** — нюанс, без которого можно работать.

## 2026-04-22 — нет поля для фиксации причины отмены заказа

- **Severity:** medium
- **Источник:** `orders_order`, сценарий отмены
- **Проблема:** при переходе заказа в `cancelled` невозможно сохранить причину (отказ клиента / недоступность товара / логистическая проблема / дубль заказа). Это затрудняет аналитику причин отмен.
- **Сценарий:** клиент отменил заказ, сказал «передумал». Оператор хочет зафиксировать причину, но поля нет. Пришлось писать в отдельный excel.
- **Предложение:** добавить поле `cancellation_reason` в `orders_order` — либо свободный TEXT, либо enum со значениями (customer_refusal / product_unavailable / logistics_issue / duplicate / other).
- **Статус:** open
- **Связанные решения:** — (требуется ADR)

## 2026-05-05 — Уникальные товары не имеют места в order_item

- **Severity:** medium
- **Источник:** `orders_order_item` / сценарий создания нового заказа (G19 калькулятор)
- **Проблема:** в проекте есть два **разных** типа товаров с разной семантикой жизненного цикла:
  - **Каталожные** — регулярно продаваемая номенклатура (Veritas, Lie-Nielsen, Infinity и т.д.). Ведутся в `catalog_product`, имеют листинги и историю цен.
  - **Уникальные** — единоразовая позиция, которую сознательно НЕ нужно заносить в каталог: товар с eBay (один лот), конкретная книга, штучный антиквариат. Повторно их не купить, и заведение в каталог только засорит его «мёртвыми» SKU.

  Сейчас `orders_order_item.product_id` — FK на `catalog_product`. Для уникального товара негде хранить inline-описание (название, вес для расчёта shipping, цена закупки в валюте, ссылка на лот).
- **Сценарий:** оператор делает калькулятор нового заказа (G19). В заказе клиента — два каталожных товара Veritas и одна уникальная книга «Hayward — Cabinetmaking and Joinery» с eBay за $35. Книгу нельзя добавить в каталог (это конкретный лот, не SKU), но и положить в заказ нельзя — нет полей.
- **Предложение:** обсудить варианты в отдельном мини-ADR:
  - **(A)** `order_item.product_id NULLABLE` + поля `inline_name TEXT NULL`, `inline_weight_kg NUMERIC NULL`, `inline_purchase_price_fcy NUMERIC NULL`, `inline_purchase_currency CHAR(3) NULL`, `inline_source_url TEXT NULL`. CHECK constraint: либо `product_id IS NOT NULL`, либо все `inline_*` заполнены.
  - **(B)** Отдельная таблица `orders_unique_purchase` (id, name, weight_kg, purchase_price_fcy, purchase_currency, source_url, notes) + FK от `order_item.unique_purchase_id` (тоже nullable). Чище семантически, дороже по стоимости.
  - **(C)** Заводить в каталоге с флагом `is_unique=true` или категорией «штучные», скрывать из обычного поиска. Минус: каталог всё равно растёт мёртвыми SKU.
- **Статус:** open — ждёт ADR
- **Связанные решения:** G19 (калькулятор нового заказа), ADR-018 (решение по архитектуре калькулятора будет ссылаться на это решение).
