## Системный промт операционного Cowork

Если ты — операционный Cowork-чат (помогаешь оператору с CRM через MCP-tools, **не** архитектурный Cowork-чат и **не** Claude Code), у тебя есть **обязательный первый шаг** до любого ответа оператору:

### Обязательный bootstrap (исполнить дословно, до первого ответа)

1. **Вызови Read** на `/Users/protey/pili-crm/docs/cowork-system-prompt.md` — весь файл, без offset.
2. **Вызови Read** на `/Users/protey/pili-crm/docs/inbox_measurement.md` — весь файл (там шапка с эвристиками сбора).
3. Только после этого отвечай оператору.

**Почему дословно:** упоминание промта в CLAUDE.md без его реального чтения = ты работаешь по дефолтному поведению, без правила двух подтверждений (раздел 3 промта), без русских меток (раздел 5), без аудита (раздел 8) и без сбора измерений для G18 (раздел 11). Все эти правила нужны оператору **в этой сессии**.

**Что не считается выполненным bootstrap:**
- «Из контекста CLAUDE.md я знаю про разделы X/Y» — нет, это знание тезисов, а не текст разделов. Прочитай файл полностью.
- «Прочитаю когда понадобится» — нет, в раздел 11 заложено, что ты сам решаешь записать сигнал в журнал; без чтения этих эвристик ты пропустишь записи.
- Чтение только частей файла — нет, формат записи и пограничные случаи лежат в нижних разделах.

**Подтверждение оператору после bootstrap (одной строкой):**
> «Промт прочитан (cowork-system-prompt.md, разделы 1–11; inbox_measurement.md, шапка). Готов работать. Раздел 11 активен до 2026-05-19, веду журнал автоматически.»

После этого работай как обычно — оператору не нужно повторно подтверждать или просить.

### Если файлы недоступны

- Если Read возвращает «file not found» — скажи оператору: «не вижу `pili-crm/docs/cowork-system-prompt.md` — нужен mount папки `pili-crm/`, иначе я работаю с устаревшими правилами».
- Если файл прочитан, но раздел 11 отсутствует и сегодня позже 2026-05-19 — bootstrap выполнен корректно, измерение уже завершилось, продолжай как обычно.

### Краткое содержание (только как ориентир — не замена чтению)

- Раздел 3 — правило двух подтверждений.
- Раздел 5 — русские метки статусов.
- Раздел 8 — три канала аудита (`IMPROVEMENTS.md` / `schema-gaps.md` / `tool-gaps.md`).
- Раздел 11 — **временный сбор измерений для G18 (ADR-016), активен до 2026-05-19**, журнал `docs/inbox_measurement.md`. Это то самое, ради чего эта секция CLAUDE.md существует.

Если ты архитектурный Cowork или Claude Code — просто игнорируй эту секцию, она не для тебя.

### Правила ведения `docs/PLAN.md`

1. **Один источник истины.** Список групп задач живёт **только** в `docs/PLAN.md`. `01_scope.md` секция «В планах» — ссылка на план, не дубль.
2. **Версионирование плана.** Шапка плана содержит историю версий. После закрытия группы или существенной правки — инкремент версии (v1, v2, ...). Снимки на дату не делаем — история живёт в git.
3. **Связь open_questions ↔ план.** Каждая запись в `06_open_questions.md` ссылается на группу `G_X` плана. При закрытии группы Архитектурный штаб проходит по зависимым open_questions и переносит их в архив.
4. **Спящие риски E-блока.** При активации (триггер сработал) — переносятся либо в новую группу плана, либо в `06_open_questions.md` (если требуется обсуждение перед реализацией).

### Чек-лист закрытия группы плана

После реализации группы `G_X` Архитектурный штаб обновляет:

- [ ] Статус группы в `docs/PLAN.md` (отметка ✅, дата, ссылка на коммит/ADR)
- [ ] `01_scope.md` секция «Сделано» — добавить запись о закрытии группы
- [ ] `06_open_questions.md` — закрыть зависимые вопросы, перенести в `_archive.md`
- [ ] `000_ADR_REGISTRY.md` — обновить дату «Последнее обновление» и HEAD; если был принят ADR — добавить в реестр
- [ ] Шапка `docs/PLAN.md` — инкремент версии плана с пометкой о закрытии

### Шаблон инициирующего сообщения для группы

Перед запуском новой группы плана нужно полное инициирующее сообщение для копирования в новый чат. В `docs/PLAN.md` для каждой группы есть **скелет** (5-7 строк). Полное сообщение разворачивается из скелета через Архитектурный штаб (Opus 4.6 + Medium, ~15-20 минут) по такому шаблону:

```
Группа: G_X — название
Цель: одно предложение, что должно получиться на выходе
Модель: Sonnet/Opus + Low/Medium/High
Префаза: какие файлы и строки прочитать первыми
Зависимости: какие группы должны быть закрыты
Ожидаемые артефакты: файлы / коммиты / ADR
Критерий готовности: CP_X из плана (с конкретной проверкой)
Чек-лист закрытия: см. выше
```

Полные инициирующие сообщения для активных в ближайшие 2 недели групп — в `docs/PLAN.md` секция «Инициирующие сообщения».

## Модель работы (где что запускать)

Долгоживущих рабочих чатов проект **не использует**. Каждая группа задач из `docs/PLAN.md` запускается в отдельном Cowork-чате, который закрывается после закрытия группы. Веб-чат claude.ai в работе не участвует — он не видит файлы репо, и в нём накапливается смешанный контекст.

| Тип задачи | Где запускать | Модель | Что делает |
|---|---|---|---|
| **Архитектурное решение** (ADR, addendum, выбор варианта) | Одноразовый Cowork-чат, имя = имя группы плана | Opus 4.6 + Medium-High | Читает префазу, обсуждает варианты, пишет ADR, обновляет реестр и план |
| **Подготовка ТЗ для Claude Code** (бывший Prompt Factory) | Одноразовый Cowork-чат, имя = имя группы плана | Sonnet 4.6 + Medium | Читает префазу + принятый ADR, формулирует детальный промт для Claude Code |
| **Реализация — код** | Claude Code (CLI) | Sonnet 4.6 (по умолчанию) | Получает промт, пишет код, тесты, миграции, коммитит |
| **Operator workflow** (apply, review, link через MCP) | Cowork-чат | Sonnet 4.6 (Cowork) | Использует MCP-tools для повседневной работы оператора |
| **Простая правка** (гигиена документации, короткий промт для CC) | Cowork-чат | Sonnet 4.6 + Low-Medium | Без отдельной ступени Prompt Factory |

**Когда не нужна отдельная ступень Prompt Factory:** простые правки (одна-две функции, документация), которые Claude Code пишет по короткому промту прямо из архитектурного Cowork-чата.

**Когда нужна:** реализация ADR с миграцией БД, новыми тестами, интеграцией; разбиение крупного ADR на несколько подзадач (G6 finance ledger — несколько пакетов).
